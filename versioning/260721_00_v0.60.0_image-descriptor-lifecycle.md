# Jupiscan v0.60.0 — versioned image-descriptor lifecycle and bounded duplicate evidence

- Date: 2026-07-21
- Change package: `260721_00`
- Scope: Android-native image metadata persistence, comparison precision, Room migration safety,
  backfill progress and duplicate-cache compatibility

## Goal / authoritative requirement

Continue strengthening the metadata created and persisted on the phone so indexing and duplicate
review are more accurate without introducing a second database, large model dependency or
functional regression. Commit, push and deploy the verified Android changes directly through the
existing Jupiscan release pipeline.

The attached desktop VISOR/Python specification remains research input. This release implements a
bounded Android-native evidence-hardening tranche; it does not claim Torch/FAISS/CLIP, crop/local
feature matching, semantic matching, a Python plugin runtime or a nine-layer VISOR system.

## Root causes addressed

1. Image rows had dHash/pHash/aHash but no producer version, so a changed algorithm could compare
   incompatible old and new values or fail to requeue stale rows.
2. The v9→v10 migration added packed image geometry as NULL, but the old backlog and readiness
   queries ignored missing geometry. Upgraded galleries could therefore report descriptor coverage
   complete while the aspect-ratio veto was inactive.
3. Production accepted `dHash <= 8` plus either pHash or aHash confirmation. A catastrophic
   disagreement in the third family was unbounded and could be hidden by the OR gate.
4. Eight one-byte LSH bands only guarantee a shared unchanged band for distance below eight, not at
   the public inclusive distance-8 boundary.
5. An unordered LIMIT page containing 100 retryable images could starve every later photo.
6. StorageReadiness observed only `index_state`; targeted descriptor updates did not necessarily
   emit new coverage.
7. The code removed destructive fallback but registered migrations only from v4, although v1-v3
   database versions had shipped. A sufficiently old direct upgrade could fail Room open.
8. Cold duplicate snapshots could resurrect image groups created by the previous decision contract.
9. Per-image descriptor UPDATE mellett a live readiness aggregate minden képnél újrafuthatott; a
   futási cap csak a sikeres írást számolta, és a keyset cursor mögötti concurrent sort nem ellenőrizte.

## Implemented contract

### Minimal persisted metadata

- Room v11 adds one `perceptualVersion INTEGER NOT NULL DEFAULT 0` column to the existing
  `file_index` row. There is no side table, path duplication, new B-tree index, model file or
  variable-size feature store.
- `PerceptualHash.CURRENT_DESCRIPTOR_VERSION` is the single producer contract.
- A comparison-ready image must be IMAGE type, current version, non-sentinel dHash+pHash+aHash and
  valid packed geometry. Current-version UNHASHABLE stacks remain terminal attempted rows.
- The legacy dHash-only writer clears pHash/aHash/geometry and stamps version 0, so it cannot create
  a mixed descriptor that appears current.
- Byte/type identity preservation copies `perceptualVersion` through browse refresh, survey upsert,
  single-file index and subtree move/rename paths; identity/type change resets it with every derived
  field.

### Lossless migrations

- Reconstructed explicit v1→v2, v2→v3 and v3→v4 migrations preserve old rows while adding
  lifecycle, dHash and structural columns.
- v10→v11 is additive and performs no user-file IO. Complete v10 image stacks with geometry, plus
  terminal all-UNHASHABLE stacks, are marked current. Missing-geometry legacy rows stay version 0
  and are requeued; full/quick content hashes and all non-image descriptors remain untouched.
- Production and tests share one ordered `allMigrations()` source of truth through v11.
- Room schema export is enabled and `app/schemas/.../11.json` is tracked as the authoritative schema
  contract from v11 forward.

### Comparison and candidate precision

- `VisualMatchEvidence` exposes descriptor completeness, geometry compatibility, three Hamming
  distances, fused score and final boolean decision.
- Every production image lookup/group call still uses `isSamePicture`, which now delegates to the
  same evidence function used by tests.
- The dHash gate and pHash/aHash confirmation remain, but the weighted three-family score must also
  pass. A narrowly bounded smooth-image fallback requires exceptionally close dHash plus one family
  and forbids a third-family distance above 32.
- A separate low-contrast recovery path extends dHash only from 8 to 10 when pHash and aHash are
  each at most 4 bits apart and the fused score is at most 6. It cannot hide a third-family conflict.
- Candidate rows are restricted to current complete IMAGE descriptors; unrelated file types cannot
  enter the photo comparison set.
- Variable-width bands cover the inclusive distance-8 contract and the bounded distance-10
  high-consensus candidate cap. Final decisions remain budgeted, fail-closed, stacked and
  complete-link verified; LSH is candidate generation only.
- pHash uses one precomputed 8×32 DCT cosine basis instead of recalculating 16,384 cosine values per
  image. The stored hash format is unchanged.

### Backfill, readiness and cache

- Backlog/readiness predicates accept only all-three-UNHASHABLE terminal rows or all-three-valid
  current rows with geometry; every mixed sentinel state is requeued.
- Keyset pagination (`ORDER BY path`, `path > afterPath`) lets one run continue beyond an entirely
  retryable first page. The 50k cap counts every visited row, the final authoritative recount catches
  rows inserted behind the cursor, and failed rows remain missing for WorkManager retry.
- Worker and arrival catch-up persist each decoded 100-row page through one short Room transaction,
  so the live readiness aggregate receives one invalidation per page rather than per image.
- StorageReadiness combines live `index_state` and a Room-observed descriptor aggregate, so a
  targeted fingerprint update emits coverage without a synthetic state write.
- Duplicate snapshot header v3 rejects the prior image-decision cache. A fresh scan repopulates it
  under the current evidence contract.

## Non-regression invariants

- Exact duplicate identity remains full current-byte content hash only.
- Similar images remain review-only and cannot authorize automatic deletion.
- Complete-link grouping, keeper protection, size filters, ordering, Select all/Deselect all and
  Recycle Bin behavior remain unchanged.
- Content/quick hashes, media vectors, text/archive descriptors, generation/checkpoint state,
  notification outbox and idempotency are preserved by v10→v11.
- No destructive Room fallback, second metadata database, Python runtime, OpenCV, ML model or new
  network/service dependency is introduced.
- Existing application/package/database identifiers remain upgrade-stable; user-visible branding
  remains Jupiscan.
- GitHub Actions artifact/release display names use Jupiscan; stable tags, APK filenames and the
  repository URL remain unchanged for download compatibility.

## Remaining risk / explicit non-goals

- Rotation/mirror/crop/local-feature invariance is not claimed in v0.60. D4/crop work needs its own
  versioned descriptor and a measured positive/hard-negative corpus before changing stored meaning.
- No 40k physical-device p50/p95/p99 or Samsung corpus benchmark was run. The code removes per-image
  cosine recalculation and starvation, but no unmeasured latency percentile is claimed.
- Readiness currently counts terminal UNHASHABLE attempts as completed coverage, while they remain
  excluded from matches. A later UI may split attempted/current/usable coverage more explicitly.
- Do not downgrade a v11 installation to a v10 APK. A forward rollback must retain schema v11 and
  disable only the current producer/matcher behavior.
