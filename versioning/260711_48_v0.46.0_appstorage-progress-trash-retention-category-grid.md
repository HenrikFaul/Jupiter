## Fixes / changes

### (1) App-storage scan — parallel, determinate progress, self-updating
- **Parallel measurement** (`AppStorageSource.queryStream`, now a `channelFlow`): the per-package
  `queryStatsForPackage` (a binder round-trip — the source of the 9–15 s sequential scan) now runs
  with bounded parallelism (`SCAN_CONCURRENCY = 8` via a `Semaphore`), cutting wall-clock
  several-fold.
- **Determinate progress**: `AppStorageOverview` gained `scannedCount`/`totalCount` (+ a `progress`
  float). The stream emits an immediate `0/N` frame, then a monotonic progress snapshot at the first
  5 packages and every 15 after (emitted under a lock so parallel workers can't reorder progress),
  then the final complete result. The screen shows a real `LinearProgressIndicator` + "X / N apps
  (P%)" in both the full-screen `ScanningView` and the in-list `ScanningBanner`.
- **"Only shows on touch" fixed**: the root cause was frame-scheduling starvation — the heavy,
  sparse-emitting sequential loop kept the Choreographer from drawing until an input event forced a
  frame. The faster parallel scan + steady, frequent progress emissions keep the UI actively
  recomposing, so the result appears on its own. Progress frames are conflated
  (`buffer(1, DROP_OLDEST)`) so a slow collector never lags; the final frame is a suspending `send`.
- **Off-main icon load**: `AppRow` now resolves the launcher icon via `produceState` on
  `Dispatchers.IO` instead of decoding the drawable on the composition thread, so the list's first
  frame doesn't jank.

### (2) Recycle-Bin auto-delete (retention)
The Recycle-Bin viewer + data layer already shipped (More → "Recycle Bin", `Destination.Trash`). New:
- **Setting** (`SettingsDataStore.trashAutoDeleteDays`, Int; 0 = OFF/default): a "Recycle Bin"
  Settings section with an "Open Recycle Bin" shortcut and an "Auto-delete trashed items" radio
  selector (Never / 7 / 15 / 30 / 60 days), wired through `SettingsViewModel` (13th combine flow) →
  `SettingsUiState`.
- **Purge** (`TrashDao.getOlderThan` + `TrashRepository.purgeOlderThan(cutoff)`): permanently deletes
  items trashed before the cutoff (row + on-disk payload), and cleans up a row whose payload already
  vanished. A row whose physical delete fails is left in place (retried next sweep) — never orphaned.
- **Worker** (`TrashPurgeWorker` @HiltWorker + `TrashScheduler`): a daily periodic sweep (battery-not-
  low), scheduled from `JupiterApp.onCreate`; it no-ops while the setting is Never. Changing the
  setting to a positive window also kicks a prompt one-time purge so it takes effect immediately.
- **Countdown**: each Recycle-Bin card shows "Auto-deletes in N days" when retention is on.

### (3) Home category grid
- A colourful 4-column `CategoryGrid` on Home (Photos / Videos / Audio / Documents / APKs / Archives
  / Downloads), each tile = colour-tinted icon badge + label + total size, tap →
  `Destination.CategoryBrowse.create(category)`. Sizes come from the already-loaded
  `HomeUiState.categories` (no new computation); colour/icon reuse the analytics palette. No

- **`TrashPurgeTest`** (Robolectric + in-memory Room + real temp files): `purgeOlderThan(cutoff)`
  deletes strictly-older items (row + payload), keeps the boundary/newer ones, cleans up a
  payload-gone "ghost" row, and purges nothing when all items are newer.
- **`AppStorageOverviewResolveTest`** (unchanged, green): the streaming display rule.
- The parallel scan + Compose progress and the category grid are device-verified (StorageStatsManager
  /AppOps and Compose UI); the new progress fields are pure data.

## Remaining risk / rollback

- Low–moderate (broad but additive). The scan rewrite keeps the same emission contract plus progress;
  the parallel workers are bounded and cancellation-aware; progress is emitted under a lock. The
  retention feature is OFF by default (no auto-deletion without an explicit opt-in), and the purge
  only ever removes a row once its payload is gone (NO DATA LOSS preserved). The category grid is a
  pure additive Home section. Rollback = revert.

## Lessons captured this iteration

- **"UI updates only on touch" is usually frame starvation, not a state bug.** A long background job
  that emits rarely lets the Choreographer skip drawing until an input event. Steady, frequent state
  emissions (and finishing the job faster) keep recomposition live without any touch.
- **Parallelize binder/IO fan-out.** Hundreds of independent `queryStatsForPackage` calls are latency-
  bound; bounded concurrency turns a 10 s serial scan into a second or two.
- **A retention feature must default to OFF and delete only verified-gone payloads** — auto-deletion
  is the one place a file manager can silently lose data, so it needs an explicit opt-in and the same
  "row dropped only after the bytes are gone" discipline as manual permanent-delete.
