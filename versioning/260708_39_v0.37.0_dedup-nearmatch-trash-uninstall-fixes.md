# v0.37.0 — Three reported bugs fixed: near-dup fires against un-backfilled originals, trashed/stale duplicates no longer error on open, App-storage refreshes after uninstall

**Date:** 2026-07-08
**Change package:** 260708_39
**Version:** 0.37.0
**Scope:** Jupiter Android app development (not managed device files).

## Reported bugs (user, with screenshots)

1. **Perceptual near-dup never fires in practice.** Two images of the same woman — one on the
   phone for months, one just downloaded/recompressed — are NOT auto-flagged as similar.
2. **Opening a duplicated file errors.** Preview shows `Not found: …/Android/.Trash/com.sec.android.app.myfiles/…`
   — a stale Samsung My Files trash path in the index that no longer exists on disk.
3. **Uninstalling an app from App-storage doesn't refresh.** The uninstalled app lingers in the list.

## Root causes

1. `FileIndexRepositoryImpl.findNearDuplicateImages` compared a newly-arrived image ONLY against
   rows that were ALREADY perceptually fingerprinted (`dao.allPerceptualHashes`). A months-old
   original the background backfill had not yet reached carries no dHash, so it was invisible and no
   SIMILAR alert could ever fire — the exact common case the user hit. (Contrast the EXACT path,
   which already hashes same-size candidates on demand.)
2. Samsung parks deleted files under `Android/.Trash/com.sec.android.app.myfiles/...` (capital
   `.Trash`), which was NOT in the index exclusions, so those volatile paths were indexed and later
   surfaced as duplicates; opening one after the bin was emptied hit a non-existent path.
3. `AppStorageScreen` only re-queried on resume when `permissionRequired && hasUsageAccess()` — so
   returning from the system uninstall dialog did not refresh the list.

## Fixes

- **Bug 1 (data/index):** `DuplicateDetector.onFileArrived` now calls `ensureImagesFingerprinted()`
  immediately before the near-duplicate lookup. It drains the whole "missing perceptual hash" work
  list in bounded batches (100), computing + persisting each fingerprint (marking undecodable files
  `UNHASHABLE` so the list always shrinks and a corrupt image can't loop). Progress persists per row
  via the identity-preserving upsert, so the first image arrival pays a one-time cost and every later
  arrival finds an empty backlog. This guarantees the comparison set is complete, so an un-backfilled
  original is matched.
- **Bug 2 (data/index):**
  - `MediaStoreIndexSource` and `IndexingWorker` exclude a new `.trash` segment (case-insensitive;
    `IndexingWorker.isExcluded` now lowercases both sides so vendor-case dirs like `Android/.Trash`
    match). `.trashed`/`.thumbnails`/`android/data`/`android/obb` unchanged.
  - The dedup READ surfaces (`findContentDuplicates`, `findNearDuplicateImages`, `duplicateGroups`)
    run results through `existingOrPruned` / an inline trash filter: any result path that is in a
    trash/recycle-bin/thumbnail segment OR whose file no longer exists on disk is excluded AND its
    row is deleted from the index. So an alert or duplicates list never points at a trashed/vanished
    file, and rows indexed before the exclusion existed self-heal on the next read — no full re-survey
    needed.
- **Bug 3 (feature/apps):** `AppStorageScreen` re-queries on every RESUME except the first (the
  ViewModel `init` load covers the first), so returning from the uninstall dialog immediately drops
  the uninstalled package (the query enumerates only installed packages) — and granting Usage-access
  still refreshes as before.

## Verification (real, in CI)

- **`DuplicateDetectorImageTest`** (Robolectric, `@GraphicsMode(NATIVE)` — real decode, no emulator):
  a metadata-only (un-fingerprinted) PNG original + a same-scene JPEG copy at a different
  resolution/quality → `onFileArrived` returns a `SIMILAR` alert pointing at the original, and the
  original is fingerprinted afterwards (backlog drained). An inverted (different) picture → no alert.
  Asserts the original genuinely starts un-fingerprinted, so the test fails without the fix.
- **`DedupResultPruningTest`** (Robolectric + in-memory Room): a byte-identical copy in a Samsung
  `Android/.Trash/...` dir is not surfaced and its row is pruned; a duplicate whose file was deleted
  is not surfaced and its row is pruned; a live duplicate is still surfaced.
- Existing `DuplicateDetectorTest` (EXACT), `PerceptualHashSourceTest`, reconciler + fusion tests
  unchanged and still pass. CI `testDebugUnitTest` is the authoritative gate.

## Remaining risk / rollback

- Low, additive. Bug 1's on-demand drain is bounded, cancellable, best-effort (never throws) and
  self-terminating once the library is fingerprinted. Bug 2's read-time pruning only deletes rows for
  files that are genuinely gone or in an excluded dir. Bug 3 adds one query per resume. Rollback =
  revert.

## Lessons captured this iteration

- **A "matches only against already-computed state" lookup silently degrades to "never matches" when
  the state is lazily populated.** The EXACT path already learned this (hash same-size candidates on
  demand); the perceptual path had the identical latent bug. When a detector depends on a background
  backfill, make the detector able to complete the backfill for its own comparison set on demand,
  rather than trusting the schedule.
- **Index rows outlive the files and the exclusion rules that were current when they were written.**
  Filter (and self-heal) at read time, not only at write time, so a policy change or a
  moved/trashed/deleted file can't surface a dead entry until the next full survey.
