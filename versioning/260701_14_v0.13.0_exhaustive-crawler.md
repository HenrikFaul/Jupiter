## Goal / why

NON-NEGOTIABLE user command: the dynamic layer must exhaustively drive the app ÔÇö **every function, every screen
variation, every feature** ÔÇö clicking through the whole app, filling forms, toggling everything, "if it takes a
thousand screens, then a thousand". The previous crawler was capped at 18 states and did one tap per step Ôćĺ only
8ÔÇô13 screens. Unacceptable, since screen analysis underpins the spec-conformance, UX/visual and Play-listing checks.

## Fixed / rebuilt ÔÇö `layers/dynamic_apk.py`

The shallow `_explore_u2` + `_tap_next_clickable` were replaced by a **systematic, deep crawler**:

- **Full per-screen action enumeration** (`_extract_actions`): every interactable node is turned into an ordered
  action ÔÇö **click** (every clickable), **edit** (every input field, filled with context-aware sample data:
  email/phone/PIN/OTP/URL/search/passwordÔÇŽ), **toggle** (every checkbox/switch), **scroll** (every scrollable list,
  swiped to reveal more), plus per-screen **open nav-drawer** and **open menu** gestures.
- **Deep DFS with back-tracking**: each state's untried actions are exhausted before backing out; the crawler
  relaunches the app if it exits, dedupes states by hierarchy signature, and stops when the reachable UI is
  exhausted or a budget is hit.
- **Permission handling** (`_dismiss_dialogs`, `_grant_all_perms`): all declared runtime permissions are granted up
  front, and permission/system dialogs ("Allow" / "While using the app" / permissioncontroller buttons) are clicked
  so flows are not blocked.
- **Activity sweep** (`_activity_sweep`): every declared activity is directly `am start`-ed to reach screens the
  interactive crawl may miss ÔÇö a coverage guarantee.
- **High budgets** (config `dynamic`): `max_states` 300, `max_actions` 2500, `crawl_time_budget_s` 720,
  `report_screens_cap` 80, plus `fill_forms` / `handle_permissions` / `activity_sweep`. CLI `--crawl-budget SEC`
  and `--max-states N` expose control.
- **Coverage metrics + finding**: the exploration dict now carries screens / actions / inputsFilled / togglesFlipped
  / scrolls / activitiesVisited-of-total / activitySweep / budgetSpentS / capped. A new **DYN-COVERAGE** finding
  reports the thoroughness (and honestly notes login-gated / server flows may be unreachable + that budget can be
  raised). `_explore_adb` fallback also enlarged (bigger monkey sweeps).

## Risk / rollback

Medium: a deep crawl takes longer (minutes) and can, on a huge app, hit the caps ÔÇö all budgets are configurable and
the crawl is time-bounded so it always terminates. Rollback = restore the 18-state single-tap loop.
