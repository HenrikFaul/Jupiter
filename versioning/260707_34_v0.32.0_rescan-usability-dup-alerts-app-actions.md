# v0.32.0 ‚Äî Index stays usable during rescans; duplicate alerts actually fire; tappable app rows

**Date:** 2026-07-07
**Change package:** 260707_34
**Version:** 0.32.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

Three user reports (with screenshots):
1. During a manual rebuild the Analytics screen collapsed from "120.2 GB analyzed" to
   "3.0 GB analyzed / Still scanning‚Ä¶", and the App-storage estimate ballooned to "~212.6 GB"
   ‚Äî read as "nothing was implemented".
2. Standing request: new files must be indexed immediately and a "you already have this"
   NOTIFICATION must fire when a duplicate arrives (re-download etc.).
3. The App-storage rows show sizes but nothing can be opened/deleted from there.

## Root causes and fixes

- **Rescan degradation.** Read gates used `isMetadataComplete()` (status == COMPLETE); a
  RUNNING rebuild is not COMPLETE, so screens fell back to a partial live walk even though the
  prior complete generation's rows were fully intact (the stale-sweep only runs on success).
  Fix: new `IndexStateRepository.isUsable()` ‚Äî COMPLETE **or** `lastCompleteGeneration > 0` ‚Äî
  adopted by the four read gates (Search, Cleanup, Analytics VM, Analytics repo). Scheduling
  decisions (JupiterApp auto-start, GrantSurveyPolicy) intentionally keep `isMetadataComplete()`.
  A first-ever partial scan and a reset index remain unusable. The absurd mid-scan estimate is
  additionally gated by `estimateReliable` (index-served or finished walk) in the UI.
- **Duplicate alerts never fired (#5 closed).** The pipeline existed end-to-end
  (`DownloadIndexObserver` observes all of MediaStore, indexes deltas, posts a notification on
  content duplicates) but `findContentDuplicates` searched only rows with a STORED hash ‚Äî and
  surveys index metadata only, so the pre-existing original never had one. The common case
  (re-downloaded file) therefore never alerted. Fix: hash-on-demand of same-size, unhashed
  candidates (exact duplicates must match in byte size, so the prefilter keeps it cheap)
  before the hash lookup, via the existing identity-guarded `hashForEntry`.
- **Tappable app rows.** Rows open a bottom sheet: system App-info (the only sanctioned way to
  clear ANOTHER app's cache/data ‚Äî platform rule), Browse `Android/media/<pkg>` in Jupiter when
  that folder exists (readable app media, e.g. WhatsApp), and Uninstall (system dialog,
  non-system apps). The sheet states the platform limit honestly.

## Adversarial review ‚Äî 2 confirmed findings fixed pre-commit

- **HIGH:** no minimum-size floor ‚Äî a zero-byte arriving file (a just-started download's
  placeholder) would have matched the constant empty-content hash of every empty file on the
  volume and spammed false alerts. Fixed with `MIN_ALERT_SIZE_BYTES` (4 KiB, matching the
  cleanup dedup floor) + test coverage.
- **LOW (race):** `hashForEntry`/`putHash` wrote back a whole-row upsert built from a
  pre-hash snapshot; concurrently a RUNNING survey may have re-stamped the row's
  `lastSeenGeneration`, so the write-back reverted the stamp and the end-of-scan sweep would
  delete a live file's row. Fixed with a targeted `FileIndexDao.updateHash` UPDATE that never
  touches the generation column + a sweep-survival test.

## Verification (real, in CI)

- **`indexStaysUsableDuringRescanOverPriorCompleteGeneration`** (Robolectric): no-state ‚Üí
  unusable; first partial scan ‚Üí unusable; complete ‚Üí usable; rescan RUNNING ‚Üí NOT complete but
  usable; reset ‚Üí unusable.
- **`newFileDuplicateIsFoundEvenWhenOriginalWasNeverHashed`** (Robolectric, REAL temp files):
  original indexed metadata-only (hash null) + same-size different-content decoy; the arriving
  byte-identical copy finds exactly the original ‚Äî the notification precondition, proven.
  Extended with the size-floor case: empty files never alert.
- **`hashBackfillPreservesGenerationStamp`** (Robolectric): `updateHash` leaves the survey's
  generation stamp intact and the row survives the sweep.
- **CI (authoritative):** to be confirmed green after push.

## Deliberately still outstanding (transparent ‚Äî NOT claimed done)

- **Perceptual near-duplicates (#13)** ‚Äî same image in a different format/resolution
  (screenshots, re-encodes). Design for next round: 64-bit dHash computed for NEW images on
  arrival + a low-priority backfill worker for existing images (thumbnail-decode), Hamming
  distance compare, alert threshold ‚â§ 10. Current round covers byte-identical duplicates.
- On-device validation of the three flows; index backlog otherwise unchanged.

## Remaining risk / rollback

- Low-medium. `isUsable` widens read gates ‚Äî the risk of serving a slightly stale index during
  a rebuild is deliberate and bounded (deltas keep it live; the rebuild replaces it on
  completion). Hash-on-demand adds IO on new-file arrival proportional to same-size candidates
  (rare). Rollback = revert this commit.

## Lessons captured this iteration

- **"Complete" and "usable" are different predicates.** Gating reads on scan-completeness made
  every rebuild LOOK like data loss. Reads should gate on data validity (prior complete
  generation intact), scheduling on completeness.
- **A feature is not shipped until its common case fires.** The duplicate alert worked only
  when the original happened to be hashed ‚Äî i.e., almost never. Trace the trigger path from
  the USER's scenario (re-download), not from the code's happy path.
