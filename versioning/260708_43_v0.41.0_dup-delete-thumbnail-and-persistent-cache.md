# v0.41.0 — Duplicate cleanup: no random thumbnail after delete + disk-persistent instant re-open

**Date:** 2026-07-08
**Change package:** 260708_43
**Version:** 0.41.0
**Scope:** Jupiter Android app development (not managed device files).

## Reported bugs (user, with screenshots)

1. **CRITICAL:** After deleting a duplicate, instead of the row disappearing, an unrelated ("random")
   image is shown in its place — "as if it must put an image there". Fix immediately.
2. Re-opening the screen STILL shows a multi-second blank white screen (the v0.40.0 in-memory cache
   does not survive the user closing/killing the app).

## Root causes

1. The duplicate file rows were rendered positionally inside a `Column` (`forEachIndexed`) with **no
   stable key**. When a delete removed a file and the rows below shifted up, Compose REUSED a row's
   composable node — including its Coil `AsyncImage` — for a *different* file, and the `AsyncImage`
   kept showing the previous file's cached thumbnail until the new one decoded. So a deleted row
   looked like it was replaced by an unrelated image. (Not a data bug — `deleteSelected` only filters
   groups, never inserts.)
2. `DuplicateScanCache` was in-memory only (`@Singleton`), so a full process death (app closed) lost
   it and the next cold start ran a full scan → blank screen again.

## Fixes

- **Bug 1 (feature/cleanup):** each duplicate row is wrapped in `key(file.path) { … }`, giving every
  file a stable node identity so a row's node/`AsyncImage` is never reused across different files.
  Belt-and-braces: the Coil `ImageRequest` now sets `memoryCacheKey(file.path)` and
  `diskCacheKey(file.path)` so a thumbnail can never be confused with another file's.
- **Bug 2 (feature/cleanup):** `DuplicateScanCache` now also persists a small on-disk snapshot of the
  last analysis (`filesDir/duplicate_scan_snapshot.tsv`, one tab-separated line per file, written via
  a temp-file + atomic rename so a crash mid-write can't corrupt it). `load()` returns the in-memory
  copy or reads the snapshot on the IO dispatcher; the ViewModel `init` loads it asynchronously and
  renders it immediately, then runs the silent background rescan. A full app kill + re-open now shows
  the list instantly. A synchronous loading state during the brief disk read prevents an
  empty-view flash. Only the group list is persisted (enough to render + act on); quality labels are
  re-probed in the background.

## Verification

- Compile + unit tests in CI. Both fixes are Compose-rendering / persistence behavior confirmed on a
  device: delete a duplicate → the row disappears cleanly, no stray image; kill the app and re-open
  the screen → the previous list appears immediately with no blank wait. No unit test is added
  because the change is inseparable from Compose node recycling + real file IO / process lifecycle,
  which the emulator-less harness does not meaningfully exercise; the snapshot serializer is a simple,
  self-contained round-trip.

## Remaining risk / rollback

- Low. Row keying is a standard Compose correctness fix. The snapshot is additive, best-effort
  (`runCatching`), written atomically, and never authoritative — the silent rescan and the
  existence-checked trash move keep the live data correct even if the snapshot is stale. Rollback =
  revert.

## Lessons captured this iteration

- **Every list row that owns async content needs a stable key.** Positional rendering of rows that
  contain a Coil `AsyncImage` is a latent "wrong image after the list mutates" bug: the recycled node
  shows the previous occupant's image. `key(stableId)` (plus path-stable cache keys) is the fix.
- **"Instant on re-open" means surviving process death, not just navigation.** An in-memory singleton
  covers back-and-forth within a session but not the user closing the app; a tiny disk snapshot that
  the ViewModel reads on init is what actually removes the cold-start blank screen.
