# Jupiter v0.55.0 — resilient duplicate-arrival alerts and delivery recovery

Date: 2026-07-13

Change package: `260713_02`

Branch: `main`

Scope: existing Jupiter Android file manager; real-device duplicate-arrival detection, notification delivery, OEM observation, persistent index migration, UI guidance and regression evidence.

## Goal / authoritative requirement

The user re-tested on a phone and reported that downloading the same image already present many times still produced no notification. The supplied core technical requirement requires a persistent, type-aware dedup engine that observes new files promptly, compares them against the warm index, survives process death and Android/OEM limits, and does not silently lose a file at an incomplete descriptor or delivery step.

Non-negotiable constraints: do not rebuild the application, do not regress existing functionality or direct names, work on synchronized `main`, and update changelog/versioning/lessons.

## Root-cause matrix and design decisions

| Failure mode | Prior behavior | v0.55 contract |
|---|---|---|
| v0.54 upgrade race | First generation-aware pass quietly baselined a file downloaded after the legacy `_ID` checkpoint. | Drain the legacy `_ID` gap fully; write generation version/baseline only after the drain proves complete. |
| Transient file/hash/provider failure | `null` represented both unique and temporarily unreadable; a cursor could advance. | `ArrivalInspectionResult.Unique/Alerted/Retry`; `Retry` aborts the batch before checkpoint advance. |
| MediaStore error | A failed query/probe could look like an empty delta or Android 10 fallback. | Android 11+ version/generation/query error is retryable and propagated to `CoroutineWorker.Result.retry()`. |
| Common OEM timing | Only ContentObserver + delayed WorkManager; a provider may lag or the process may be killed. | ContentObserver + bounded common-folder FileObserver/direct stable inspection + signal-time durable WorkManager backup + foreground/periodic reconciliation. |
| Notification blocked | Canonical decision inserted before Android silently rejected notification; future alert was suppressed. | Decision and delivery are separate durable lifecycles; blocked/failed decisions remain retryable and are collapsed into a summary once delivery becomes possible. |

## Acceptance criteria

- [x] A byte-identical newly arriving file is indexed then exact-hash compared against same-size indexed candidates, including metadata-only originals.
- [x] Pending→finalized MediaStore updates, legacy-upgrade backlog and equal-generation page boundaries cannot silently skip an arrival.
- [x] A temporary read/decode/provider/Room failure cannot be treated as a unique file or move the durable delta cursor beyond it.
- [x] Live process signals receive an immediate direct path and an event-time persistent backup; stopped processes converge by WorkManager/foreground/periodic reconciliation.
- [x] Permission/app/channel notification blocks persist as undelivered, not as a falsely delivered decision; later retry is atomic and idempotent.
- [x] A user can navigate from Duplicate cleanup to Android notification settings without changing selections or delete behavior.
- [x] Known Room v4→v5→v6→v7 upgrades preserve the persistent dedup/index data; no destructive migration fallback is reintroduced.

## Remaining risk / rollback

- Android cannot promise immediate cold-process wake-up for arbitrary third-party filesystem events; the direct observer optimizes alive-process latency, while WorkManager/foreground/periodic passes provide convergence. Doze/OEM policy can delay, not silently cursor-skip, a transient arrival.
- There is no physical Samsung + Chrome full instrumentation run in this workspace. API 34 runtime evidence and JVM/Robolectric proof do not replace that device-specific release gate.
- `FileObserver` is deliberately bounded to common user-write trees (1,024 watchers) to avoid unbounded inotify/battery cost. Less common paths remain covered by MediaStore and authoritative reconciliation.
- Rollback is isolated to the new outbox/publisher, observer safety layer and v6→v7 additive schema fields; downgrading an installed Room v7 app is not supported, as normal Android version rollback is not a data-preservation path.

## Lessons captured

- An idempotent business decision must not be confused with an actually delivered OS notification.
- A delta checkpoint is only safe after provider query success and a terminal per-item processing outcome; temporary I/O must remain observable/retryable.
- A fast observer needs a durable event-time backup before any stability debounce to be process-death safe.
