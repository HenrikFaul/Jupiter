## Root cause of the broken drag (diagnosed, then fixed)

The transfer drag was wired with `detectDragGesturesAfterLongPress` on the whole row, but `FileRow`'s
internal `combinedClickable` also consumes the long-press тАФ the two competed, so the drag almost never
started, and there was no visible affordance. **Fix:** a dedicated **drag handle**.

## Verification

- **The live-index verifier caught a real bug I had committed too early:** `childPathsUnder` used `LIKE`
  without `ESCAPE`, so a literal `_`/`%` in a directory name acted as a wildcard and over-purged sibling
  folders' index rows (index-only, no file loss). **Fixed** by adding `ESCAPE '\'` and escaping the
  prefix (`escapeLike`). Deleting `photos_2024` no longer touches `photosX2024`.

## Lessons captured this iteration

- **SQL `LIKE` needs `ESCAPE` whenever the pattern is built from user/filesystem data** тАФ `_` and `%` are
  wildcards and `_` is ubiquitous in real paths; a subtree match without escaping silently over-deletes.
- **Compose gesture conflicts:** a row's `combinedClickable` long-press starves
  `detectDragGesturesAfterLongPress` on the same element тАФ use a dedicated handle with `detectDragGestures`.
- **Don't commit before the workflow's verify phase reports** тАФ the escape bug would have shipped; the
  data-loss/correctness verifier is worth waiting for. (Sync to `тАж/lessons-system/lessons_orchestrator.md`.)
