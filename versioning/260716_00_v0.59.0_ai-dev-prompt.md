# AI development record — Jupiscan v0.59.0 compact metadata index

## Objective

Extend and persist the existing file metadata so it materially strengthens indexing and duplicate detection while minimizing Room/SQLite storage on the phone, with no functional regression.

## Authoritative user requirement

“Folytasd a fájlok metaadatainak elkészítésével és a metaadatainak mentésével úgy, hogy ezek a duplikációkeresést és indexálást minél jobban támogassák, és ennek a metaadatbázisnak a létrehozása és telefonra elkészítése minimális tárhelyet foglaljon.”

## Implementation constraints

- Existing Jupiscan app; do not create a new application or second index.
- Work directly on synchronized `main`; preserve unrelated local files.
- Do not weaken exact identity, type-aware near comparison, complete-link grouping, keeper protection, selection scope or Recycle Bin safety.
- Do not truncate cryptographic evidence merely to save storage.
- Metadata must be lazy, bounded and derived from already-required IO/decode where possible.
- Legacy schema/data must migrate losslessly without destructive fallback or whole-file rescan.
- Derived descriptors must be invalidated with their source byte/type identity.
- Local Gradle, emulator/device and GitHub Actions are independent evidence sources; claim only what actually ran.
- Update version, changelog, summary, this AI record and the external lessons bible.

## Files to inspect before editing

- `changelog.md`, `README.md`, `versioning/VERSIONING_CONTROL.md`, chronological version records.
- External `lessons_orchestrator.md`, required common/Android lessons, especially duplicate fingerprint lifecycle lessons.
- `FileIndexEntry`, DAO, database, DI migrations, repository and index state machine.
- Full/quick/perceptual/structural/media fingerprint producers, matchers, backfill workers and arrival detector.
- Existing migration, hash cascade, image group and media matcher tests.

## Exact execution steps

1. Confirm `main`, fast-forward pull from `origin/main`, and preserve untracked owner files.
2. Trace every persisted duplicate evidence field from producer through Room to exact/near consumers.
3. Compare normalized side-table versus compact same-row storage and select the lower-overhead representation.
4. Add raw SHA-1 and bounded ordered-vector codecs with legacy dual-read and malformed-input failure.
5. Add nullable compact columns and a production v9→v10 migration that converts only provably valid values.
6. Move the hot exact lookup index from legacy TEXT to compact BLOB.
7. Capture image/video geometry from existing decode/metadata passes and use it only as a conservative Similar veto.
8. Separate byte-identity retention from type-specific descriptor retention.
9. Atomically invalidate every stale derived field when current bytes change.
10. Defer WAL checkpoint/VACUUM/optimize to versioned retryable WorkManager maintenance outside the migration transaction.
11. Add byte-size, roundtrip, actual migration, persistence, invalidation and negative matching tests.
12. Run compile, targeted tests, full tests, lint, APK assembly, emulator open/maintenance proof and whitespace checks.
13. Update documentation and external lesson, commit/push `main`, then observe the concrete GitHub Actions run.

## Files expected to change / be added

Changes are confined to file-index metadata, duplicate descriptor producers/matchers, Room migration/maintenance, their tests, build version, What's New and required documentation. Navigation routes, public screen signatures, package/application ID, signing identity, Storage Truth, file operations, Vault, Remote, Relay and Automation must not be changed.

## Verification

- Codec: 40-hex SHA-1 → exact 20-byte roundtrip; ordered longs preserve value/order; malformed input fails closed.
- Migration: seed a real v9 SQLite schema with valid and invalid legacy rows, run production `MIGRATION_9_10`, verify conversion/preservation/indexes.
- Repository: BLOB write/read/index lookup, compact media descriptor roundtrip, type-change invalidation and current-byte refresh invalidation.
- Matcher: geometry-positive resize/rotation and negative incompatible-shape cases while stacked/multi-frame evidence remains mandatory.
- Full unit suite, lint and APK build.
- Emulator: retained-data install where available, app process launch, database schema header, compaction marker and fatal log audit. State explicitly whether an actual seeded v9 device upgrade was or was not proven.
- Remote: concrete implementation commit, push and GitHub Actions run id/conclusion; never infer CI success from local Gradle.
