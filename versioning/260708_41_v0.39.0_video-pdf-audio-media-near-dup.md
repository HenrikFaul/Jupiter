# v0.39.0 — Video / PDF / audio near-duplicate extractors, wired LIVE behind a testable seam

**Date:** 2026-07-08
**Change package:** 260708_41
**Version:** 0.39.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

The user asked to finally build the video / PDF / audio near-duplicate extractors. These are the last
media layers from the dedup blueprint. Their decoders (`MediaMetadataRetriever` / `PdfRenderer` /
`MediaCodec`) require genuine on-device codecs the emulator-less CI does not have, so the honest
engineering shape is: put the decoders behind a **seam** (`MediaFingerprintSource`), prove the entire
arrival-pipeline wiring in CI against a scripted fake, and verify the actual decode on a device. This
ships all three layers live and keeps the "no unverifiable claims" rule.

## Added

- **`MediaFingerprintSource`** (interface / seam) + **`AndroidMediaFingerprintSource`** (real, Hilt
  `@Binds`):
  - `videoKeyframeHash` — `MediaMetadataRetriever` samples a representative frame near mid-duration
    (`OPTION_CLOSEST_SYNC`), reduced to a 64-bit dHash.
  - `pdfRenderHash` — `PdfRenderer` renders page 0 (white backing, aspect-preserving, ≤320px) to a
    bitmap → dHash.
  - `audioAcousticHash` — `MediaExtractor` + `MediaCodec` decode ≤12 s of PCM, down-mix to mono, and
    reduce to a 64-window **loudness-envelope** fingerprint (relative window energy → volume-invariant).
  - Contract identical to the other sources: real fingerprint on success, `StructuralHash.UNHASHABLE`
    when decoded-but-not-comparable (marked once, never retried/matched), `null` on transient failure.
- **`BitmapDHash`** — the shared bitmap→dHash reduction now used by images, video frames, and PDF
  pages; `PerceptualHashSource` was refactored onto it (one validated implementation).
- Repository `findNearDuplicateVideo/Pdf/Audio` over a shared `nearByStructuralHamming` helper
  (Hamming ≤ per-type threshold within the type, existence-pruned); `structuralHash` backfill scope
  widened to VIDEO/PDF/AUDIO. Per-type thresholds: video ≤ 10, PDF ≤ 8, audio ≤ 10.
- `DuplicateDetector` VIDEO/PDF/AUDIO branches (shared `mediaNearCheck`) after exact + image +
  text/archive; on-demand `ensureStructuralFingerprinted` now fingerprints the media types too, so an
  original that predates the feature is still in the comparison set.

## Verification (real, in CI — `testDebugUnitTest`)

- **`DuplicateDetectorMediaTest`** (Robolectric + in-memory Room, scripted `FakeMediaFingerprintSource`):
  a metadata-only (un-fingerprinted) video/PDF/audio original + a within-threshold copy → SIMILAR alert
  pointing at the original (the on-demand backfill fingerprints the original first); an out-of-threshold
  file → no alert; two undecodable (UNHASHABLE) files never match despite sharing the sentinel.
- Existing `StructuralFingerprintSourceTest`, `DuplicateDetectorStructuralTest`, `DuplicateDetectorImageTest`,
  `PerceptualHashSourceTest` (after the `BitmapDHash` refactor) and the reconciler/scorer tests are
  unchanged and still pass.

## Honest boundary (device-verified, NOT CI-verified)

The three `AndroidMediaFingerprintSource` decode methods run real platform codecs that Robolectric/JVM
cannot execute, so their correctness is verified by installing the APK and observing the alerts on a
device — not claimed from a green CI. CI proves the seam, the per-type comparison, the thresholds, the
backfill, the alerts, and the existence-pruning. This is the deliberate split; the alternative (a
Robolectric shadow returning stub bitmaps) would prove nothing about real decoding.

## Deliberately still outstanding

- On-device semantic **embeddings** (CLIP-class image/text/frame-sequence) — the only remaining
  blueprint extractor; needs a bundled/loaded model. Plugs into the same seam + fusion scorer.
- Video temporal fingerprint (multi-frame) — today one representative keyframe; a multi-frame
  signature would cut false positives on videos that share a mid-frame. Threshold kept conservative.

## Remaining risk / rollback

- Moderate, well-isolated. Media decode is confined behind the seam and each method is total
  (try/catch → UNHASHABLE/null, all handles closed in `finally`); a decode failure degrades to "no
  media fingerprint," never a crash. No schema change (reuses `structuralHash`, DB v4). The image path
  is unchanged behaviourally (same dHash math, now via `BitmapDHash`, still covered by its test).
  Rollback = revert.

## Lessons captured this iteration

- **Seam the untestable, prove the testable.** When correctness depends on a runtime the CI lacks
  (real codecs), isolating it behind an interface lets the entire pipeline — the part that actually
  had the bugs historically — be proven with a fake, while the irreducible device dependency is
  labelled device-verified instead of faked green.
- **One reduction, many producers.** Extracting `BitmapDHash` meant video frames and PDF pages inherit
  the exact, already-validated image dHash math for free — a re-encode of any of them lands at the same
  small Hamming distance, and there is a single place to get wrong.
