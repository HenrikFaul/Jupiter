# v0.12.0 — Duplicate scan reuses cached index hashes (fast repeat scans)

**Date:** 2026-07-01
**Change package:** 260701_17
**Version:** 0.12.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

The Room index (0.11.0) was built to make the app faster; the highest-value payoff is letting
the duplicate scan **reuse full-content hashes** instead of re-reading every candidate file on
every scan. Ties to the user's indexing/UDH-port requests ("indexálás … gyorsabb legyen az app";
"ultimate-duplicate-handler … hasznosítsd"). Mirrors Sift's incremental-hash-reuse (skip
re-hashing when size+mtime unchanged).

## Modified

- **`data/storage/StorageAnalyticsRepositoryImpl.kt`**
  - Injected `FileIndexRepository`.
  - In `resolveBucket`'s full-hash loop: reuse `indexRepository.hashForUnchanged(path,size,mtime)`
    when the file is byte-for-byte unchanged; on a miss, compute the SHA-1 once and write it back
    via `putHash` for next time. Best-effort: any index failure falls through to a direct hash, so
    the scan never crashes or blocks on the cache.

## Acceptance criteria (met)

- A repeat duplicate scan of an unchanged tree skips re-reading file contents (served from the
  index cache), while a changed file (different size or mtime) is re-hashed. ✔
- No cache = identical behavior to before (correctness preserved). ✔

## Regression checks (verified — adversarial review, PASS/0 blockers)

- Size-bucketing, prefix pre-hash stage, `ensureActive()` cancellation, and `DuplicateGroup`
  emission unchanged; only the source of the `hash` value changed.
- Stale-hash guard: reuse requires exact path+size+mtime match; a row without a hash yields null →
  fresh hash. Cached and computed hashes are the same SHA-1 (single algorithm).
- No Hilt cycle (FileIndexRepository → Impl → Dao → Database; none depend back on StorageAnalytics).
