# v0.47.0 — Home category sizes (APKs/Archives no longer 0 B) + perceptual image duplicates in cleanup

**Date:** 2026-07-11
**Change package:** 260711_49
**Version:** 0.47.0
**Scope:** Jupiter Android app development (not managed device files).

## Reported items (user, with screenshots) — TWO regressions/gaps

1. **APKs and Archives read 0 B on the Home "Categories" grid**, even though the browser lists 48
   APKs (1.7 GB) and 31 archives (236 MB). Photos/Videos/Audio/Documents/Downloads were correct.
2. **Duplicate IMAGES never appear in Duplicate cleanup**, despite hundreds of duplicate photos —
   while duplicate videos and text files DO appear. The perceptual (dHash) image comparison built in
   earlier cycles is nowhere to be found in the app.

## Root causes

1. **Single-bucket vs type-based categorization.** The Home tiles used the storage-analytics
   `categoryForPath`, which applies a **Downloads-path heuristic FIRST**
   (`StorageAnalyticsRepositoryImpl.categoryForPath` → `if (isInDownloads(path)) return DOWNLOADS`).
   Every downloaded `.apk`/`.zip` therefore counted as DOWNLOADS (the 24.4 GB tile), leaving APPS and
   ARCHIVES at 0. The category BROWSER is type-based (MediaStore), orthogonal to folder, so it
   correctly listed them — hence the mismatch.
2. **Perceptual image dedup was never wired to the cleanup UI.** The cleanup list is produced solely
   by `StorageAnalyticsRepository.findDuplicates` → `duplicateGroups`, which groups by **exact SHA-1
   content hash** only. The dHash near-duplicate IMAGE detector (`findNearDuplicateImages`) had ONE
   caller — `DuplicateDetector.onFileArrived` — feeding an arrival notification + an `_alerts` Flow
   that **no screen observes** (`observeDuplicateAlerts` had zero collectors). So re-sized/re-encoded
   photos (byte-different → different SHA-1) never grouped. The video "same/best quality" labels are
   just `MediaQualityProbe` decorations on exact-duplicate groups, not near-dup detection — so images
   weren't special-cased out; they simply were never eligible.

## Fixes / changes

### (1) Type-based Home category sizes
- `MediaStoreCategorySource.summarize(category): CategoryUsage` — a lean projection (path + size)
  cursor pass that sums the SAME type-based set the browser lists (excluding recycle-bin paths), so
  a tile shows exactly what opening the category reveals. An `.apk` in Download counts toward BOTH
  APKs and Downloads, matching the browse.
- `HomeViewModel.refresh()` now builds the grid `categories` from `summarize()` across the 7 tile
  categories (in parallel) instead of the analytics single-bucket overview — and no longer walks the
  whole filesystem, so Home loads faster. (The storage-analytics pie is unchanged — it is a
  deliberately different single-bucket view; only the reported Home tiles were wrong.)

### (2) Perceptual image near-duplicates in the cleanup list
- `FileIndexRepository.nearDuplicateImageGroups(threshold)` — clusters fingerprinted images by dHash
  Hamming distance via union-find: pass 1 groups exact hashes (O(n), resolution/compression-robust),
  pass 2 merges distinct hashes within `threshold` (bounded by `MAX_NEAR_PAIR_COMPARISONS = 50M` so a
  huge all-unique library never stalls — it falls back to exact grouping). Groups are `similar = true`,
  files ordered largest-first, existence-pruned.
- `DuplicatesViewModel` merges these image groups into the list when the exact scan completes,
  dropping any image already shown in an exact group (no double-listing), and kicks the perceptual
  backfill each scan so coverage grows.
- `DuplicateGroup.similar` flag; the cleanup card shows "N similar photos" for these groups.

## Verification (real, in CI)

- **`NearDuplicateImageGroupTest`** (Robolectric + in-memory Room + real temp files): identical and
  2-bits-off dHashes cluster together; a separate identical pair is its own cluster; hashes >8 bits
  apart stay singletons (dropped); groups are `similar` and largest-first; empty when no near
  neighbour exists.
- `AppStorageOverviewResolveTest`, `TrashPurgeTest`, `StorageExclusionsTest`,
  `FileRepositorySearchExclusionTest` unchanged and green.
- The category-tile sizes (MediaStore) and the on-screen "similar photos" groups are device-verified.

## Remaining risk / rollback

- Low–moderate. The category change swaps the tile data source (type-based, matching the browser) and
  removes a filesystem walk from Home; the image-dedup addition is additive (exact groups unchanged;
  image groups appended with de-dup against exact paths) and coverage depends on the perceptual
  backfill, which the scan advances. The near merge is bounded. Rollback = revert.

## Lessons captured this iteration

- **A "size" tile must use the same source as the list it links to.** Surfacing the analytics
  single-bucket sizes next to a type-based browser produced a glaring 0 B where dozens of files exist.
- **A great algorithm that isn't wired to a surface the user can act on is invisible.** The perceptual
  dHash detector worked, but only fed a notification Flow nothing observed — so users never saw their
  photo duplicates. Integration into the cleanup list is what makes it real.
