## Goal / why

The user re-reviewed the full dedup-engine blueprint prompt and asked to verify what is actually done
and to fill the gaps. Gap analysis (below) shows the 20-section design is fully documented and much is
shipped, but only two near-duplicate layers ran LIVE in the arrival pipeline: identity (all types) and
perceptual (images). This iteration ships the two remaining layers that are **verifiable in pure-JVM
CI without an emulator or media decode** — text/code and archives — end-to-end and tested, and marks
them shipped in the blueprint. Video/PDF/audio/embeddings remain explicit backlog (they need on-device
decode/ML we will not fake as "done").

## Added (this iteration)

- **`StructuralFingerprintSource`** (`data/index`): pure-JVM extractor with
  - `textSimHash(path)` — reads ≤1 MiB, computes the formatting-insensitive `TextSimHash`; binary
    (NUL-containing) or empty content → `TextSimHash.EMPTY` (stored, never matches); IO error → null.
  - `archiveTreeHash(path)` — opens a ZIP-family container and FNV-1a-hashes the sorted
    `(name, uncompressed-size, crc)` member set; non-ZIP/empty → `StructuralHash.UNHASHABLE`
    (conservative, never matches); IO error → null. Recognises the same files repacked with different
    compression (different bytes) as the same contents.
- **`FileIndexEntry.structuralHash`** column (DB v3→v4, destructive fallback — the index is a cache),
  compared only within a file type so the shared column is unambiguous. Preserved across surveys when
  identity (size+mtime) is unchanged, exactly like `perceptualHash`/`contentHash`.
- **DAO / repository**: `updateStructuralHash`, `structuralHashesOfTypes` (excludes the UNHASHABLE
  sentinel), `filesMissingStructuralHash`; `putStructuralHash`, `findNearDuplicateText` (Hamming ≤ 3),
  `findSameArchiveContents` (equality), `filesNeedingStructuralHash` — all existence-pruned (a
  trashed/vanished file is never surfaced and its row is pruned, matching v0.37.0).
- **`DuplicateDetector`** new branches after exact + image: CODE → SimHash near-check; ARCHIVE/APK →
  member-tree same-contents check. Each first drains its own missing-fingerprint backlog on demand
  (bounded, cancellable, progress-guaranteed — same pattern as image fingerprinting), so an original
  that arrived before the feature shipped is still in the comparison set. Every alert is fused through
  `SimilarityScorer` for a tier + explanation. The SIMILAR notification now uses the alert's own
  type-specific explanation, so image/text/archive all read correctly.

## Verification (real, in CI — the authoritative `testDebugUnitTest` gate)

- **`StructuralFingerprintSourceTest`** (pure JVM): reformatted code is near / unrelated code is far /
  binary is EMPTY; identical members at different compression levels produce the SAME fingerprint,
  different members differ, a non-ZIP file is UNHASHABLE; missing file → null.
- **`DuplicateDetectorStructuralTest`** (Robolectric + in-memory Room): a metadata-only (un-fingerprinted)
  code original + a reformatted copy → SIMILAR alert pointing at the original; a repacked (recompressed)
  archive → SIMILAR "same contents" alert; unrelated code/archives → no alert.
- Existing detector/reconciler/scorer/perceptual tests unchanged (constructor updated for the new
  dependency) and still pass.

## Deliberately still outstanding (transparent — NOT claimed done)

- Video keyframe, PDF rendered-page, and audio chromaprint extractors, and on-device embeddings
  (§4.6 true semantic). These require `MediaMetadataRetriever`/`PdfRenderer`/ML Kit / a decode runtime
  that this emulator-less CI cannot exercise, so shipping them would be an unverifiable claim. The
  fusion scorer, tiers, explainability, and the `StructuralFingerprintSource` on-demand-backfill
  pattern are ready to receive them with no re-architecture.
- Wiring `SampleOffsets` into a live large-binary prefilter and an on-device APK extractor for
  `ApkComparator` (both pure pieces already exist and are tested).
- The normalized multi-entity Room schema (§11) — the single `file_index` table carries all
  fingerprints today; splitting into descriptor/score/decision tables is a future migration.

## Remaining risk / rollback

- Moderate-low. The DB version bump uses destructive fallback (index rebuilds from storage). The new
  detector branches only run for CODE / ARCHIVE / APK arrivals and are guarded/best-effort (the whole
  `onFileArrived` is wrapped in `runCatching`). Existence-pruning matches the shipped v0.37.0 contract.
  Rollback = revert (the column is additive and defaulted null).

## Lessons captured this iteration

- **"Designed" and "documented" are not "live."** A blueprint marking a layer shipped is worthless if
  the arrival detector never calls it. The honest unit of completion is *runs end-to-end and is proven
  by a test that fails without it* — which is why this change ships only the layers pure-JVM CI can
  actually exercise, and leaves media/ML extractors explicitly unshipped rather than claimed.
- **One backfill pattern, reused.** The image "fingerprint the whole missing backlog on demand before
  comparing" pattern generalised verbatim to text and archives, so an original that predates a feature
  is always comparable — the same latent "compares only against already-computed state" bug that hit
  perceptual near-dup is pre-empted for every new layer.
