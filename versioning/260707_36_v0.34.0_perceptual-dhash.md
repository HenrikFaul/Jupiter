# v0.34.0 — Perceptual (dHash) near-duplicate detection (#13 delivered)

**Date:** 2026-07-07
**Change package:** 260707_36
**Version:** 0.34.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

Explicit repeated user demand: detect that a newly-arrived image is the SAME PICTURE as one
already on the device even when the bytes differ (different format, resolution,
re-compression — e.g. a messenger re-encoding a photo on download), and alert immediately.
The content-hash path (v0.32–0.33) only covers byte-identical copies.

## Design

- **`PerceptualHash` (pure core):** 64-bit difference hash over a 9×8 grayscale luminance
  grid; bit = "pixel darker than its right neighbour". Structure survives re-encode/resize,
  so same-picture pairs land at Hamming distance 0–6; unrelated images average ~32.
  Threshold: ≤ 8/64. `UNHASHABLE` (Long.MIN_VALUE) marks tried-but-undecodable files and is
  excluded from every comparison (never matches, not even itself). Android-free → fully
  JVM-tested.
- **`PerceptualHashSource` (decoder):** bounds-only decode → subsampled decode
  (`inSampleSize` toward 64px) → one bilinear 9×8 scale → Rec.601 luma. A few ms per photo.
  OOM → UNHASHABLE; unexpected transient errors → null (retried later).
- **Arrival path (`DownloadIndexObserver`):** exact byte-duplicate check first (unchanged);
  if none and the file is an image: compute + persist the fingerprint, compare against the
  full fingerprint set (lean `(path, hash)` projection scanned in memory — trivial at tens
  of thousands of longs), and post a distinct **"Similar image detected"** notification.
- **Backfill (`PerceptualHashBackfillWorker`):** batches of 100, ≤ 2000 per run,
  battery-not-low, cooperative cancellation. Progress is guaranteed: every attempted row is
  marked (hash or UNHASHABLE), so the work list strictly shrinks and corrupt images cannot
  loop. Retry continues when the per-run cap is hit. Kicked from app start, every
  foreground, and after every successful survey (chained in `IndexingWorker`).
- **Fingerprint survival:** `perceptualHash` follows the same identity rule as
  `contentHash` on ALL rewrite paths (`writePreservingHashes`, `indexFile`,
  `replaceChildren`, `onMovedOrRenamed` root; descendants carry it via `entry.copy`).
  Without this, every 12 h survey would have nulled all fingerprints and re-decoded the
  entire photo library twice a day — caught during design, encoded in a Room test.
- **Storage:** new nullable `perceptualHash` column; DB version 2 → 3 with the documented
  destructive fallback (the index is a cache; it rebuilds once after the update and the
  state row resets with it, so "complete" can never point at wiped data).

## Verification (real, in CI)

- **`PerceptualHashTest` (pure JVM):** rising gradient = all 64 bits / falling = 0;
  determinism; single-sample perturbation flips ≤ 2 bits and stays near; inverted structure
  = distance 64, not near; UNHASHABLE never near (incl. itself); grid-size validation.
- **`PerceptualHashSourceTest` (Robolectric, native graphics):** the user-facing contract —
  the same scene saved as PNG 240×180 and JPEG 120×90 q70 IS near; the inverted scene is
  NOT; a text file named `.jpg` is UNHASHABLE.
- **`IndexStateMachineTest` additions (Robolectric + Room):** fingerprint survives a rescan
  of an unchanged image and is cleared when identity changes; near-duplicate lookup returns
  only within-threshold matches, excluding self / far / UNHASHABLE rows.
- **CI (authoritative):** to be confirmed green after push.

## Deliberately still outstanding (transparent — NOT claimed done)

- Heavily cropped/rotated derivatives (e.g. a screenshot of a photo including status bar)
  can exceed the threshold — a known dHash limitation, documented in-app expectationally.
  Rotation-invariant matching would need multi-orientation hashing (possible follow-up).
- No UI browsing of "similar groups" yet — detection is alert-driven; a Duplicates-screen
  section for perceptual groups is a natural next round.
- On-device validation of the full loop (backfill completion on ~40k images, then a
  re-shared image alerting).

## Remaining risk / rollback

- Medium-low. DB version bump wipes the cache-index once (rebuilds automatically; the
  survey is now reliable). The backfill is bounded, battery-gated, and self-terminating.
  The in-memory comparison set is ~16 bytes × images (trivial). Rollback = revert this
  commit (DB downgrade also destructive — acceptable for a cache).

## Lessons captured this iteration

- **New column on a rewrite-heavy table? Audit every writer.** Four separate code paths
  rebuilt whole rows; any one of them would have silently wiped the new column on its next
  pass. The preservation rule must ship WITH the column, encoded in a test.
- **Sentinel for "tried and failed" is what makes a backfill terminate.** Null-means-retry
  alone turns one corrupt file into an infinite loop.
