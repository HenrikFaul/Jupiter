## Added / Modified

- **`data/index/FileIndexRepositoryImpl.kt` — duplicate-delete safety.** `hashForEntry` now stats the
  file and trusts the stored `contentHash` ONLY when the current size AND mtime still match the indexed
  identity. On mismatch it re-hashes the CURRENT bytes and refreshes the row (size/mtime/hash);
  vanished / non-regular files return null (never grouped). This makes the index duplicate path as safe
  as a live walk — two genuinely-different files can never be grouped from a stale hash.
- **`data/file/FileRepositoryImpl.kt` — Browser↔index.** `listFiles` self-heals the index on every
  successful disk read (`replaceChildren(path, raw)` — the RAW unfiltered listing, so hidden files are
  not pruned; stale children are pruned). New `listFromIndex(path, sort, filter)` returns indexed
  children with the same sort/filter (empty on any failure).
- **`domain/repository/FileRepository.kt`** — `listFromIndex` added to the interface.
- **`feature/browser/FileBrowserViewModel.kt`** — on a disk-read Failure, fall back to
  `listFromIndex`; when the index has the directory, render its last-known contents (updating the
  query cache/signature/breadcrumbs/tabs consistently) instead of an empty error.
- **`feature/analytics/*`** — `StorageAnalyticsUiState` gains `fromIndex`+`indexedCount`; the VM injects
  `FileIndexRepository` and sets them from `isPopulated()`/`stats()`; the screen shows a compact
  "Instant from index · N" chip (parity with the Cleanup banner).
- **`app/build.gradle.kts`** — `versionName` → 0.23.0.

## Regression checks (verified)

- `listFromIndex` empty on any failure; disk stays authoritative for the visible listing (fallback
  only on Failure). Self-heal is best-effort (`runCatching`) and never affects the returned result.
- New DI injection (`FileIndexRepository` into `StorageAnalyticsViewModel`) — binding exists
  (`IndexModule`); no cycle. `FileRepositoryImpl` already injected it in v0.22.0.
- Brace/paren balance verified on all 7 edited files.
- **Adversarial review workflow** (`review-index-round2`, run `wf_0739ab9d-dfd`): six independent lenses
  (deletion-safety, correctness, Room/SQL, coroutines, Compose/state, regression/DI) with per-finding
  skeptic verification. **Five findings confirmed and fixed** in a follow-up commit before merge:
  1. **Hash-cache wipe** — the self-heal `replaceChildren` nulled every child's precomputed content
     hash on each browse (breaking downloads-dedup `byHash` + forcing re-hash). Fixed: `replaceChildren`
     now preserves a stored hash when the child's size+mtime are unchanged (single `childEntries` query).
  2. **Empty `NOT IN ()`** — `replaceChildren(path, [])` generated invalid SQL. Fixed: empty listing →
     `deleteByParent`.
  3. **Overview cross-volume pollution** — `overviewFromIndex` aggregated ALL indexed rows (incl. a
     browsed SD card) into the primary-volume overview. Fixed: scope to `volume.rootPath`.
  4. **Inline self-heal latency** — the index write was awaited inside `listFiles`, delaying the
     listing. Fixed: fire-and-forget on a repository `indexScope`.
  5. **Transient-empty wipe** — an empty disk read (possible transient failure) would prune the index.
     Fixed: only self-heal on a NON-EMPTY listing.
  6. **Survey-excluded dirs via browse** (workflow-confirmed, low) — browsing `.thumbnails`/`.trashed`/
     `Android/data|obb` self-healed those files into the index, so index-served duplicate/large-file
     lists could surface (and offer to delete) files the live scan excludes. Fixed: the self-heal skips
     survey-excluded directories (`isIndexableDir`), mirroring `IndexingWorker`'s exclusions.
  Deferred (low, edge-case): browsing filesystem root `/` (a parentPath edge on a mostly-inaccessible
  path) — rare and low-harm.
  Final workflow tally (`wf_0739ab9d-dfd`, 21 agents): 4 confirmed findings, all fixed (1–3 in b64b34a,
  #6 here); the verifiers corroborated that the hardened `hashForEntry` (null on a vanished file) already
  neutralises the duplicate-scan consequence.

## Lessons captured this iteration

- **On a delete-driving path, an index must be treated as a hint, not truth.** Re-verify the file's
  current identity (size+mtime) before trusting a cached hash; re-hash on mismatch. Cheap (a stat per
  size-collision candidate) versus catastrophic (deleting a file wrongly grouped as a duplicate).
- **Self-healing beats one-shot surveys.** Letting ordinary browsing persist each listing keeps the
  index complete between full rebuilds, so the index-served screens converge to accuracy through use.
