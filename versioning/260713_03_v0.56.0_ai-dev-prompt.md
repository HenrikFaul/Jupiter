# Jupiter v0.56.0 — implementation prompt / continuation contract

## Objective

Repair two observed Android file-manager defects without functional regression:

1. Similar-photo detection must not show zero merely because the perceptual descriptor backfill failed transiently or was never scheduled.
2. A raw 256 GiB filesystem allocation must display as the device’s retail 256 GB capacity, not 274.9 GB.

## Mandatory inputs and working rules

- Work only from up-to-date `main`: `git pull --ff-only origin main` before changes; commit and push directly to `main` after verification.
- Read `README.md`, `changelog.md`, versioning documents from oldest to newest, relevant Forge lessons, and this prompt before implementation.
- Preserve exact full-hash deduplication, keeper protection, visible-scope selection, recycle-bin deletion, Automation safety and notification-outbox behavior.
- Never use destructive Room migration. Update `changelog.md` and add a detailed versioning record for every round.

## Required implementation behavior

### Perceptual images

- Model descriptor outcomes as terminal versus retryable. Only a definite decoder statement that the image is unsupported may persist `UNHASHABLE`; null, temporary IO, MediaStore/provider, Room, or decoder availability failures must retry.
- An explicit user request from Duplicate cleanup must enqueue a valid, bounded descriptor worker even when continuous indexing is disabled. This is user-requested review, not silent background mutation.
- Do not combine WorkManager expedited work with unsupported constraints. Preserve a non-expedited constrained background path and use an explicit expedited fallback path for the user action.
- On migration, requeue only the legacy erroneous image sentinel rows. Preserve all content hashes, paths, decisions, and non-image rows.
- Classify MediaStore image MIME types as images even if the filename lacks an extension.

### Storage capacity

- Use platform total/free sources. Do not sum category sizes to invent a total.
- Format user-facing capacity in decimal consumer units. If total is an exact raw binary-GiB allocation, map it to the matching retail tier (256 GiB → 256 GB). Retain platform free bytes and retain genuine nonstandard OEM values.

## Required tests / proof

- Unit-test retry-vs-terminal descriptor policy, explicit request when continuous indexing is disabled, binary 256 GiB retail conversion, and MIME-only image classification.
- Run `:app:assembleDebug :app:testDebugUnitTest :app:lintDebug --no-daemon --no-build-cache`.
- On an API 34 emulator, upgrade a v7 database with legacy sentinel rows, assert v8 migration requeues those rows, open Duplicate cleanup, and verify `PerceptualHashBackfillWorker` reaches `SUCCEEDED`.
- Inspect generated lint for modified paths, run `git diff --check`, stage only task-owned files, commit directly to main and push.

## Completion rule

Do not claim completion based solely on a card label. Completion requires a successful build/test/lint gate, real worker scheduling evidence, explicit migration evidence, changelog/versioning updates, and a pushed `main` commit. State remaining OEM/large-corpus limits accurately.
