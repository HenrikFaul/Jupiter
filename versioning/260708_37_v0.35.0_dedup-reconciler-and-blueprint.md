# v0.35.0 ŌĆö Arrival dedup actually fires (checkpoint reconciler) + full dedup blueprint

**Date:** 2026-07-08
**Change package:** 260708_37
**Version:** 0.35.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

The user reported ŌĆö repeatedly, across four release iterations ŌĆö that duplicate detection "still
doesn't work," and asked for the strong feature build-out described in a detailed dedup-engine
design prompt. Despite prior fixes (POST_NOTIFICATIONS request, identity debounce, on-demand
hashing, perceptual dHash), the alert still never fired on their device.

## Root cause (the real one this time)

Duplicate detection ran **only** through `DownloadIndexObserver`, which is alive **only while the
app process is alive**. The dominant real-world case ŌĆö a file downloaded/received/captured with
Jupiter closed ŌĆö was observed by nothing, and the next app open only ran the metadata survey (which
indexes but does not dedup new files). So the "you already have this" alert was structurally
impossible for exactly the scenario the user was testing.

Two compounding defects made it fail even while alive:
1. On many devices the `ContentObserver` fires with the bare collection URI (or no URI), so
   `resolvePath(uri)` + `moveToFirst()` resolved an **arbitrary** row, not the new file.
2. Nothing ever caught up on files that landed between process deaths.

## Fixed this iteration

- **Checkpoint-based MediaStore delta reconciliation** (`DedupReconciler`): queries MediaStore for
  every file whose `_id` is newer than a persisted checkpoint (in DataStore), runs each through the
  detector, and advances the checkpoint monotonically. Keyed on `_id` (unique + strictly
  increasing) rather than a 1-second date, so a bulk import of hundreds of files in one second
  paginates gap-free. This catches files added while the app was dead and never mis-resolves a row.
  Cheap when nothing is new, so it runs on every foreground (`MainActivity.onStart`), every observer
  signal, and the 12-hour periodic worker.
- **Baseline rule:** first run records the current max `_id` and does NOT alert on the pre-existing
  library (no alert storm); only later arrivals alert. The baseline is set ONLY when a real id is
  observed and storage access is held ŌĆö a fresh install before permission never pins a low
  checkpoint that would later re-alert the whole library.

## Adversarial review ŌĆö 3 confirmed defects fixed pre-commit (8 agents, AOSP/WorkManager-level)

- **HIGH ŌĆö baseline poisoning:** on fresh install before storage permission, `maxObservedId()`
  returned 0 and the old `coerceAtLeast(1)` pinned the checkpoint to 1, so after the grant
  `_id > 1` matched the ENTIRE library ŌåÆ retro-alert storm. Fixed: baseline is set only when a real
  id (`> 0`) is observed, else the checkpoint stays 0 and retries; and the reconciler is gated on
  `StorageAccessManager.hasAllFilesAccess()`.
- **HIGH ŌĆö expedited worker dead on API 26-30:** `DedupReconcileWorker` was `setExpedited` without a
  `getForegroundInfo()` override, so on API < 31 (WorkManager runs expedited work as a foreground
  service) it would fail and dedup would never run there. Fixed: the tiny, frequent reconcile is no
  longer expedited (plain KEEP work, runs on all API levels).
- **MEDIUM ŌĆö same-second pagination gap:** &gt; 200 files sharing one `date_added` second were
  skipped at the page boundary. Fixed by keying on the unique `_id` (no ties).
- **`DownloadIndexObserver` reduced to a pure trigger:** the fragile URI resolution and its own
  notification code are gone; it just kicks the reconciler (debounced; WorkManager KEEP coalesces
  bursts). Also kicks once on startup.
- **`DuplicateDetector`:** single authority for "a file arrived ŌåÆ index + exact (content hash) +
  perceptual (image dHash) + notify + emit alert." Both the real-time observer path (via the
  reconciler) and the catch-up path funnel through it, so behavior is identical.
- **Seams for testability:** `NewFileSource` (implemented by `MediaStoreIndexSource`) and
  `DedupCheckpointStore` (backed by `SettingsDataStore`) let the reconciler be tested end-to-end
  without a real MediaStore/DataStore.
- **`docs/DEDUP_ARCHITECTURE.md`:** the full 20-section production blueprint the user requested ŌĆö
  layered architecture, type taxonomy, 7-layer similarity model, descriptor pipeline, candidate
  retrieval, fusion scoring with per-type weight + threshold matrices, Room schema, WorkManager
  chain, performance/scaling, FP/FN mitigation, UX, observability, test matrix, anti-patterns ŌĆö
  grounded in the actual codebase and this reconciler, and marking what is shipped vs planned.

## Remaining risk / rollback

- Low-medium. The reconciler only ever widens detection (processes files the old path missed) and is
  idempotent (monotonic checkpoint, index-by-identity). The observer simplification removes fragile
  code. Worst case is a redundant reconcile run (guarded by a mutex + WorkManager KEEP). Rollback =
  revert this commit; the periodic survey and image backfill are unaffected.

## Lessons captured this iteration

- **"It doesn't work" for an event feature is usually the observation model, not the matcher.** Four
  rounds fixed the matching/notification path; the actual defect was that nothing observed the file
  at all in the common case. Trace the trigger from the USER's scenario (download while app closed),
  not from the code's happy path.
- **A ContentObserver is a signal, not a data source.** Trust it to say "something changed," then ask
  the authoritative store (MediaStore delta since a checkpoint) what actually changed.
- **A checkpoint is what makes catch-up possible across process death.** Without a durable
  high-water mark, "what's new since last time" is unanswerable after a kill.
