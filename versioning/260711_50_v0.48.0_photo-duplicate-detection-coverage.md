# v0.48.0 — Photo duplicate detection actually finds the gallery's duplicates (coverage + scale + auto-surface)

**Date:** 2026-07-11
**Change package:** 260711_50
**Version:** 0.48.0
**Scope:** Jupiter Android app development (not managed device files).

## Reported item (user, with screenshots)

The gallery is full of duplicate photos (39,635 images, obvious visual dupes), yet Duplicate cleanup
shows **zero image groups** — only 6 video/APK groups. The user (correctly) called this a core
regression: image duplicate detection was a strength and is now invisible.

## Root cause (verified end-to-end)

The photos are **near-duplicates (byte-different re-encodes/resizes), not byte-identical** — so the
exact SHA-1 path (`duplicateGroups`) legitimately never groups them, and only the perceptual dHash
path (`nearDuplicateImageGroups`, added v0.47) can. But that path only considers images that already
carry a `perceptualHash`, and:

1. **Coverage:** the ONLY proactive bulk fingerprinter, `PerceptualHashBackfillWorker`, was capped at
   **2,000 images/run** and returned `Result.retry()` under WorkManager's **exponential backoff** (up
   to 5 h). A 39,635-image library needs ~20 backoff-spaced runs → hours/days to cover. So the
   fingerprint set was essentially empty when the user opened Cleanup.
2. **No wait:** `DuplicatesViewModel` kicked the backfill (KEEP, fire-and-forget) and then called
   `nearDuplicateImageGroups` **immediately**, against the near-empty set → zero groups, silently.
3. **Scale:** even with coverage, the near-merge was O(d²) and **skipped entirely** for ~39k distinct
   hashes, leaving only exact-dHash equality (missing 1–3-bit-off re-encodes).

## Fixes / changes

### (1) Backfill converges fast — foreground + big per-run cap
`PerceptualHashBackfillWorker` now drains the whole missing-fingerprint backlog with a large per-run
cap (`MAX_PER_RUN = 20_000`, vs 2,000) and promotes itself to a **foreground service** when the
backlog is large (`FOREGROUND_THRESHOLD = 400`), like the index survey — so Doze/background limits
can't throttle it. A 40k gallery is covered in one or two short-backoff runs (minutes), not ~20
runs (days). Progress is still guaranteed (every row marked with a hash or `UNHASHABLE`), and a
stop mid-pass returns retry to resume. A "Analyzing photos for duplicates — N/M" notification shows
during the foreground pass.

### (2) Scalable near-merge — LSH banding
`FileIndexRepository.nearDuplicateImageGroups` pass 2 now uses **LSH banding** (8 one-byte bands):
two hashes within `threshold` Hamming distance share ≥1 identical band, so only same-band candidates
are compared — ~n per band instead of n². Re-encoded/resized copies whose dHash is a few bits off now
group even at 40k+ scale (pass 1 still handles identical-dHash copies). A comparison budget backstops
pathological skew.

### (3) Auto-surface as coverage grows
`DuplicatesViewModel` now keeps the exact groups and re-merges image near-dup groups on an ~8 s poll
while `imagesNeedingPerceptualHashCount() > 0`, so photo-duplicate groups appear on their own as the
backfill fingerprints the library — no manual rescan. `DuplicatesUiState.analyzingPhotos` drives an
"Analyzing your photos…" full-screen state (when no groups yet) and an in-list banner (when some
exist). New `imagesNeedingPerceptualHashCount()` on the repo/DAO; the image-group merge de-dups
against exact-group paths so nothing appears twice.

## Verification (real, in CI)

- **`NearDuplicateImageGroupTest`** (Robolectric + in-memory Room + real files), extended: identical +
  2-bit-off dHashes cluster; a separate identical pair is its own cluster; >8-bit hashes stay
  singletons; and a NEW case proves LSH banding catches a 3-bit-off near pair whose differing bits
  span three different bytes (byte0/byte3/byte7), while keeping a far image out. Empty when no near
  neighbour exists.
- `AppStorageOverviewResolveTest`, `TrashPurgeTest`, `StorageExclusionsTest`,
  `FileRepositorySearchExclusionTest` unchanged and green.
- The fingerprint decode, the foreground worker behavior, and the on-screen progressive surfacing are
  device-verified (WorkManager/decoding are device-only); the clustering is proven in CI.

## Remaining risk / rollback

- Moderate but contained. The worker runs a large foreground decode pass (one-time; fingerprints
  persist while identity is unchanged) — memory-bounded (subsampled decode), cancellation-aware,
  resumable. The LSH merge is bounded and additive. The poll is capped (≤6 min) and cancels on
  rescan. Rollback = revert.

## Lessons captured this iteration

- **A detector is only as good as its input coverage.** The dHash clustering was correct, but the
  fingerprint backfill — capped + backoff-throttled — never populated the library, so the feature was
  invisible. Making the backfill converge (foreground, big cap) is what makes detection real.
- **Don't query a background-built index and assume it's ready.** The scan read the fingerprint set
  the instant it kicked the async builder; surfacing results progressively (poll until coverage
  completes) matches how the data actually becomes available.
- **O(n²) similarity doesn't scale to a real gallery — band it.** LSH turns a 40k-image near-merge
  from "skipped" into ~n-per-band.
