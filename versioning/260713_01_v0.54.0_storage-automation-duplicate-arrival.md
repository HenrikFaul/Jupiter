## Goal and authoritative user requirement

The user reported four concrete device-visible gaps and repeated the non-regression constraint:

- the OEM storage UI showed 256 GB while Jupiter showed only about 222 GB;
- Automation did not explain its purpose and needed at least five selectable, editable, removable, activatable/suspendable and safely testable examples, all suspended by default and with file deletion forbidden;
- the duplicate hero had to count images/similar results together with exact duplicates;
- duplicate size filtering plus Select all/Deselect all had disappeared and had to return without ever selecting a keeper;
- downloading the same image again still produced no duplicate notification.

The implementation had to preserve all existing routes, public names, exact-hash safety, perceptual review, Recycle Bin behavior, keeper protection, index generation semantics and the direct-main delivery contract.

## Root-cause decisions

### Storage capacity

The old value was the `StatFs` size of the emulated user-data filesystem and was formatted in binary units. The system storage surface is allowed to show the physical media capacity advertised in retail. Android provides that display source through `StorageStatsManager.getTotalBytes(UUID_DEFAULT)` and a display-oriented free-space value through `getFreeBytes`.

Chosen design:

1. use `StorageStatsManager` total/free for the primary volume;
2. retain `StatFs` as the readable filesystem lower bound;
3. only if the system source is absent/restricted, round the filesystem lower bound to the next Android-style retail tier;
4. format volume capacity in decimal units without changing normal file-size formatting.

### Arrival notification

The old `_ID > checkpoint` cursor consumed or stepped past a download's pending MediaStore row. Finalization updated the same row, so `_ID` did not change and the completed bytes were never inspected. A leading-edge debounce could also discard the completion signal, and `ExistingWorkPolicy.KEEP` discarded a signal arriving while a reconcile was active.

Chosen design for Android 11+:

- persist and compare MediaStore version + `GENERATION_MODIFIED`;
- query only finalized `IS_PENDING = 0` rows;
- quiet-baseline on MediaStore version replacement;
- trailing-edge observer debounce;
- `APPEND_OR_REPLACE` unique work so a completion signal leaves a serialized trailing pass.

Android 10 retains the `_ID` path with pending-row exclusion; authoritative surveys remain responsible for deletes/renames/moves.

## Remaining risks and rollback

- OEMs may withhold/fail the retail-capacity service; the deterministic fallback remains an estimate based on the readable data filesystem. Rollback is isolated to `PrimaryStorageCapacityResolver` and capacity call sites.
- Android 10 lacks generation metadata; its fallback cannot provide the same update-proof cursor semantics as Android 11+. The 15-minute/full foreground survey remains the convergence path.
- Android notification permission/channel denial is a platform-visible user choice and cannot be bypassed; detection/indexing still occurs.
- Full protected system partitions cannot be enumerated by a third-party app. Jupiter accounts for their bytes without inventing paths or categories.
- Automation stays explicit/manual by design; making active rules silently schedule themselves would be a separate product/data-safety decision.

## Lessons captured

- Android retail capacity must not be inferred from a user-data `StatFs` value when `StorageStatsManager` is available.
- MediaStore arrival cursors must model pending→finalized UPDATE, not only INSERT identity.
- Automation safety must be enforced both at authoring and execution, and default examples must be one-time upgrade-safe state rather than destructive seed resurrection.
