# v0.33.0 — Uninstall works; duplicate alerts unblocked (2 root causes); background freshness

**Date:** 2026-07-07
**Change package:** 260707_35
**Version:** 0.33.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

Three on-device reports after v0.32.0:
1. Tapping an app → Uninstall does nothing.
2. (Repeated core request) The storage survey must run at FIRST open and then keep itself
   fresh in the background — never start scanning anew on each open.
3. Duplicate alert tested with 4 re-downloaded image pairs — zero notifications.

## Root causes and fixes

- **Uninstall (silent no-op).** Since Android 9 (API 28), `ACTION_DELETE` /
  `ACTION_UNINSTALL_PACKAGE` require the manifest permission **`REQUEST_DELETE_PACKAGES`**;
  without it the system rejects the request (SecurityException, swallowed by `runCatching`).
  Added (normal install-time permission) → the system uninstall dialog now appears.
- **Duplicate alerts — blocker 1: POST_NOTIFICATIONS never requested.** Android 13+ requires
  the runtime grant for ANY notification; nothing in the app requested it, so
  `notificationsAllowed()` was false and every alert — and even the indexing
  foreground-progress notification — was silently dropped. `MainActivity` now requests it at
  startup (result unused; notifications are an enhancement, never a gate).
- **Duplicate alerts — blocker 2: sliding debounce swallowed the completion event.**
  MediaStore fires repeatedly while a download is written (<1.5 s apart); the path-only
  sliding window suppressed every follow-up — including the FINAL event with the completed
  bytes, so the finished file was never hashed. The debounce is now keyed by file IDENTITY
  (path+size+mtime) via the pure `ChangeDebounce` (JVM-tested): chatter about unchanged
  bytes coalesces, the completion (new size) always processes. `IS_PENDING` MediaStore rows
  are skipped (hashing half-written bytes is wasted IO; the publish event follows).
- **Background freshness.** (a) `IndexRefreshKickWorker`: a 12-hour periodic worker
  (battery-not-low) that re-enqueues the survey via `ensureIndexed()` (KEEP). Deliberately a
  kicker — funneling through the single unique one-time work name means two surveys can
  never race generations. Scheduled from `JupiterApp` (enabled-gated, KEEP) and re-scheduled
  when indexing is re-enabled; `cancel()` now cancels it too. (b) `MainActivity.onStart`
  re-ensures the survey whenever the app foregrounds and the index has never reached
  COMPLETE (killed mid-build, access granted late) — previously only process creation did.

## Deliberately still outstanding (transparent — NOT claimed done)

- **Perceptual near-duplicates (#13):** the alert fires for byte-identical copies. If a
  shared/re-downloaded image was re-compressed by the source app, bytes differ — that needs
  the planned dHash pass (compute on arrival + background backfill, Hamming compare).
- On-device validation: grant the notification prompt on first launch of this build, then
  re-download an existing file (byte-identical) — the alert should fire.
- Whether the user's device has ever completed a full survey is not yet confirmed; the
  foreground re-ensure + periodic kicker now converge it without manual action.

## Remaining risk / rollback

- Low. New permission is install-time and Play-policy-unproblematic (uninstall is a core
  file-manager function). The periodic kicker only enqueues KEEP work. Identity-keyed
  debounce processes strictly MORE events than before (bounded by the 512-entry cap + 4 KiB
  alert floor). Rollback = revert this commit.

## Lessons captured this iteration

- **An intent that "does nothing" is usually a missing manifest permission swallowed by a
  catch-all.** Audit `runCatching { startActivity(...) }` sites when adding actions.
- **A notification feature does not exist until the runtime permission is requested.**
  Declaring POST_NOTIFICATIONS in the manifest is not requesting it.
- **Debounce on identity, not on name.** Event streams about mutating files must key on
  content identity, or the terminal event — the only one that matters — gets coalesced away.
