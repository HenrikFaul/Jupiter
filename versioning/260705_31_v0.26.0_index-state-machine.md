# v0.26.0 — Index state machine + generation sweep + Robolectric proof

**Date:** 2026-07-05
**Change package:** 260705_31
**Version:** 0.26.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

A second detailed expert review (18 defects) confirmed the index life-cycle is still broken:
completeness was decided by row count in Cleanup/Analytics (only search used the DataStore flag),
there was no generation-based global stale sweep, the subtree move wasn't transactional, and there
were no tests exercising Room. This iteration lands the **foundation** those defects share — a
proper state machine — and proves it with **Robolectric Room tests that run in CI**, honouring the
review's demand that "compiles" is not proof.

## Fixed this iteration

- **#2 — completeness in Room.** New `index_state` table (`IndexState`/`IndexStatus`
  EMPTY/RUNNING/COMPLETE/DIRTY/FAILED + activeScanGeneration/lastCompleteGeneration/timestamps/
  filesSeen/lastError), `IndexStateDao`, and `IndexStateRepository(Impl)`. The `SettingsDataStore`
  `indexComplete` flag is **removed** — completeness now lives transactionally with the data, so a
  destructive DB wipe resets it to EMPTY (→ rebuild) rather than leaving a stale "complete" flag.
- **#1 — one authority everywhere.** `CleanupViewModel`, `StorageAnalyticsRepositoryImpl` (the
  `indexIsPopulated()` gate behind overview/large-files/duplicates), `StorageAnalyticsViewModel`,
  `SearchViewModel`, and `JupiterApp` now gate on `IndexStateRepository.isMetadataComplete()`
  (status == COMPLETE), never on `isPopulated()`/row count.
- **#3 — generation scan + global stale sweep.** `FileIndexEntry.lastSeenGeneration`; the worker
  opens a generation via `beginScan()`, writes every row with `upsertScanned(items, gen)`, then on
  success runs `sweepStaleGenerations(gen)` (`DELETE … WHERE lastSeenGeneration != 0 AND < gen`) and
  `completeScan(gen)`. Deleted files (seen by a prior survey, not this one) are removed; delta/browse
  rows (generation 0) are never swept.
- **#7 (part) — atomic subtree move.** `onMovedOrRenamed` now calls a Room `@Transaction`
  `dao.replaceSubtree(oldPaths, rewritten)` (delete + insert in one transaction), so a crash between
  the delete and insert can't lose the subtree.
- **#8 (part) — real disable.** Disabling indexing cancels the unique work (`IndexingScheduler.cancel`),
  clears the DB, and resets state to EMPTY; `JupiterApp` gates the auto-build on `indexingEnabled` too,
  so it never silently restarts after the user turns it off.

## Verification (real, in CI)

- **Robolectric + in-memory Room** unit tests (`IndexStateMachineTest`, runs under `testDebugUnitTest`):
  1. partial scan is never COMPLETE (status RUNNING);
  2. completed scan is COMPLETE and sweeps a deleted file (ghost `/s/c` removed);
  3. delta rows (generation 0) survive the sweep;
  4. a rescan of an unchanged file preserves its cached hash;
  5. a directory rename rewrites the whole subtree preserving the child's hash;
  6. a prefix-sibling (`photos_2024`) is untouched by renaming `photos`.
- Added Robolectric 4.14.1 + androidx-test-core + room-testing; `testOptions.unitTests` enabled.
- Brace/paren balance + dangling-ref checks on all 20 changed Kotlin files.
- **CI (authoritative):** to be confirmed green (compile + unit + Robolectric).

## Deliberately still outstanding (transparent — NOT claimed done)

- **#4** supplemental filesystem reconciliation (MediaStore sees no directories/`.nomedia`; COMPLETE
  currently means "the survey ran", not "every byte indexed"). The fallback walk still only triggers
  on an empty MediaStore result.
- **#5** new-download duplicate detection when the existing file's hash is null; **#6** MediaStore
  generation delta-sync + version check + `DeltaSyncWorker`; **#7** full `IndexMutationCoordinator`
  (copy/restore subtree, extract/import); **#9** single rescan pipeline; **#10** foreground +
  checkpoint + resumable worker; **#11** timestamp normalisation (s vs ms); **#12** multi-volume;
  **#13** perceptual near-duplicates; **#14** FTS + Paging + size/hash DB indexes + explicit
  migrations; and the remaining end-to-end instrumentation matrix.

## Remaining risk / rollback

- Medium. The generation sweep trusts MediaStore coverage; browse-added non-MediaStore files (gen 0)
  are kept but a future full survey won't re-derive them until the #4 reconciliation lands. Robolectric
  in CI is a new moving part (downloads `android-all` at test time). Rollback = revert this range.

## Lessons captured this iteration

- **One completeness authority, checked everywhere.** Fixing only search left Cleanup/Analytics
  trusting a partial index — a gate is only as good as its least-consistent caller.
- **Put the "done" bit next to the data.** Completeness in a separate store (DataStore) can outlive a
  data wipe and lie; in-Room state resets with the data.
- **Prove Room behavior with Robolectric, not assertions in prose.** (Sync to
  `…/lessons-system/lessons_orchestrator.md`.)
