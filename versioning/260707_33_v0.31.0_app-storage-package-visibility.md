# v0.31.0 — App storage shows the REAL per-app total (package visibility + cache fix)

**Date:** 2026-07-07
**Change package:** 260707_33
**Version:** 0.31.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

User report with screenshots: the App-storage screen showed **10.4 GB across 362 apps**, while
Storage Analytics correctly estimated **~95.4 GB** of app data ("app data & caches the filesystem
can't show"). The visible list contained ONLY system apps (Google Play services, Google Speech,
Samsung Gallery/Cloud/Keyboard…) — no user-installed apps at all. The user (rightly) refused to
accept this: "other file managers solve it."

## Root cause

Three stacked defects:

1. **Package-visibility filtering (the big one).** Since Android 11 (API 30),
   `PackageManager.getInstalledApplications()` silently filters its result to packages visible to
   the caller. Without `QUERY_ALL_PACKAGES`, that means: own app + a subset of system packages.
   Exactly matching the screenshot — every listed app was a preinstalled system app; the
   storage-heavy user apps (games, messengers) were invisible, so their tens of GB never entered
   the list or the total. Other file managers hold `QUERY_ALL_PACKAGES`; file managers/device
   search are an explicitly permitted category for it under Play policy.
2. **Cache double-counting.** `StorageStats.getDataBytes()` already INCLUDES the cache;
   v0.29 summed `app + data + cache`, inflating every app by its cache size and making the row's
   "data" figure overlap the "cache" figure.
3. **Wrong volume assumption.** Every package was queried on `UUID_DEFAULT`; apps living on a
   different (adoptable) volume would report wrong/zero stats.

## Fixed this iteration

- **Manifest: `QUERY_ALL_PACKAGES`** (normal install-time permission, no prompt), with a comment
  documenting the file-manager Play-policy justification and `tools:ignore` for the lint nag.
  (Play Console declaration needed at publish time — noted below.)
- **Correct size semantics**: `AppStorageInfo.totalBytes = appBytes + dataBytes` (cache is a
  subset of data); new `dataBytesExcludingCache` for the row label so app · data · cache sum to
  the total — matching how system Settings splits the same numbers.
- **`AppStorageSource`**: per-app volume via `ApplicationInfo.storageUuid ?: UUID_DEFAULT`;
  metadata flag dropped (`getInstalledApplications(0)`).

## Adversarial review — a planned "safety net" was PROVEN WRONG and removed pre-commit

The first draft reconciled the per-app sum against `StorageStatsManager.queryStatsForUser`
(an all-apps aggregate), surfacing any excess as an "apps the system doesn't list individually"
line. A 4-lens adversarial review (with AOSP-source-level verification, 12 agents, 6 confirmed
findings) demolished it:

- **HIGH (confirmed):** the aggregate's `dataBytes` includes the user's ENTIRE shared storage —
  `installd getUserSize` measures the whole `/data/media/<user>` tree (DCIM, Download, Movies…),
  so "aggregate − per-app sum" ≈ the user's own browsable media. The header would have shown tens
  of phantom GB "in apps the system doesn't list individually" on essentially every real device —
  replacing the old under-report with a systematic over-report.
- **MEDIUM (confirmed):** the aggregate's `appBytes` walks the device-wide `/data/app` (code of
  apps installed only for OTHER users/profiles — e.g. Samsung Secure Folder) plus
  `/data/dalvik-cache`, none of which a per-app sum for this user can attribute → permanently
  positive phantom remainder on multi-profile devices.
- **MEDIUM + LOW (confirmed):** volume-scope mismatch (visible sum spans all volumes; aggregate
  only `UUID_DEFAULT`), making `max()`/difference math structurally unable to represent both
  components; plus two copy/comment inaccuracies ("cross-profile" claim; "used by N apps" wording).

Resolution: with `QUERY_ALL_PACKAGES` making the package list complete, the per-app sum IS the
honest answer. The aggregate reconciliation was deleted before commit (`AppStorageReconciler` +
its test + `unattributedBytes` + the header line), and `AppStorageSource` now documents WHY
`queryStatsForUser` must not be used as a cross-check. All six findings are thereby resolved at
the root; the two build/compile lenses had verified the surviving fixes compile and their JVM
tests pass.

## Remaining risk / rollback

- Low. The surviving change is: one manifest permission, corrected arithmetic, per-app volume
  UUID. `QUERY_ALL_PACKAGES` needs a Play-Console declaration at publish time (permitted
  category: file manager). Note: per-app and header totals can legitimately be LOWER than v0.29
  for cache-heavy apps (the double-count is gone) while the overall coverage is far higher.
  Rollback = revert this commit.

## Lessons captured this iteration

- **A silently-filtered API is worse than a failing one.** `getInstalledApplications` returned a
  plausible-looking 362-package list — all system apps — with no error. When a list's TOTAL
  contradicts an independent estimate, suspect visibility/permission filtering first.
- **Two platform numbers are only comparable if their inclusion semantics match.** The
  "obvious" cross-check (`queryStatsForUser` vs Σ`queryStatsForPackage`) measures different
  universes (shared media, other profiles' code, dexopt artifacts). The adversarial review's
  AOSP-level verification caught this BEFORE it shipped a new wrong number; the fix was to
  delete the mechanism, not to patch it.
- **Read the stats API's inclusion semantics before summing.** `dataBytes ⊇ cacheBytes`;
  summing overlapping "parts" silently inflates. Encode the invariant in a test.
- **When the user says "other apps can do it", they're usually right about a permission.**
