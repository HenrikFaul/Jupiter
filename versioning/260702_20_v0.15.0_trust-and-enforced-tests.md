# v0.15.0 — Data-transparency trust surface + enforced archive tests

**Date:** 2026-07-02
**Change package:** 260702_20
**Version:** 0.15.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

Two of the three "next" items the user approved:
1. Close the **trust** UX gap the Play-preference research surfaced (users distrust vague permission use)
   with an honest "Your data & privacy in Jupiter" surface.
2. Begin moving from *documented* rigor (the 100 verification factors) to **enforced** rigor by turning
   the two highest-priority JVM-runnable factors — Zip-Slip extraction safety and archive round-trip
   fidelity — into real unit tests that run in CI (`testDebugUnitTest`) and fail the build if broken.

## Added

- **`feature/privacy/DataTransparencyScreen.kt`** — static, honest Material3 screen: each permission +
  *why* (All-Files/READ_MEDIA/biometric/INTERNET/notifications), "everything stays on-device, no ads,
  no third-party trackers", Vault + secret encryption, analytics opt-in & OFF by default, recoverable
  Recycle-Bin deletes. Reachable from a new Settings **"Your data & privacy"** row.
- **`app/src/test/.../data/file/ArchiveZipSlipTest.kt`** — builds malicious zips (`../escape.txt`,
  `../../escape.txt`, `foo/../../escape.txt`, absolute-style entry) and asserts extraction ends
  `OperationState.FAILED` with an error and **never writes outside the extraction root**; a benign
  archive extracts to `COMPLETED` with byte-matching files.
- **`app/src/test/.../data/file/ArchiveRoundTripTest.kt`** — `createZip → extractZip` preserves bytes
  and **recreates an empty subdirectory** (re-verifies the 0.6.0 empty-dir fix).

## Regression checks (verified, 0 blockers)

- No production logic changed; `SettingsScreen` signature + all existing settings/nav/destinations
  preserved; the new screen is pure UI. Brace-balance + wiring hand-verified.
- Test correctness cross-checked against the real code: `resolveSafeEntry` throws `IOException` →
  caught → `failure()` sets `state=FAILED, errorMessage=msg`; `File(root, "/abs")` re-bases inside the
  root on the JVM; `collectZipSources` names entries relative to the root's PARENT (so the `payload/`
  prefix in the round-trip assertions is correct); empty dirs get a `/`-terminated entry that
  `extractZip` recreates via `mkdirs()`.

## Remaining risk / rollback

- Low. Tests only read the public API; a red test would signal a real regression (the point). Rollback =
  revert this commit. If `runTest` + a standalone `UnconfinedTestDispatcher` ever flags an uncompleted
  coroutine (no `delay` used, `.toList()` completes synchronously, so not expected), switch to the
  runTest-provided scheduler.
- Planned next: fold in research round-2 factors (workflow `wpirybxny`); convert more high-priority
  factors into tests (path self-overwrite guard, MediaQuality ordering, formatter/locale).

## Lessons captured this iteration

- **Verify every test assertion against the real source before trusting a green workflow** — the
  round-trip test's `payload/` prefix and the zip-slip terminal state both depended on exact behavior
  (`collectZipSources` parent-relative naming; `failure()` errorMessage) that had to be confirmed.
- **`ArchiveManager` is Android-free** (dispatcher-only), which is what makes these security/integrity
  factors enforceable as JVM unit tests without an emulator. Prefer Android-free core logic so it is
  CI-testable. (Sync to `…/lessons-system/lessons_orchestrator.md`.)
