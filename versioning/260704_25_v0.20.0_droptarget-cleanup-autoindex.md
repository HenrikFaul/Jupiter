# v0.20.0 — Dual-pane drop-target highlight, Cleanup polish, auto index survey

**Date:** 2026-07-04
**Change package:** 260704_25
**Version:** 0.20.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

User feedback (on-device screenshots) + a re-stated request:
1. **Dual pane** — while dragging a file to the other pane, dynamically tint the folder the object is
   currently over so the user sees where it will drop (live, following the finger).
2. **Smart Cleanup** — the Large-files / Duplicate-files lists should be **collapsible, collapsed by
   default**; show real **thumbnails**; and the **"Reclaim X" bar is hidden behind the system nav
   buttons** — fix the insets.
3. **Real-time index** (re-stated) — a thorough survey that persists and updates in real time on every
   file op, so opening the app never triggers a deep scan; new downloads auto-detected as content
   duplicates.

## Added / Modified

- **`feature/browser/DualPaneScreen.kt`** — `DragState.hoverTarget` (the folder path under the pointer),
  set from `computeDropFolder(pointer)` **inside `onDrag`** (recomputed every move) and cleared on
  end/cancel. The folder row tints (`isDropHover = drag.active && item.isDirectory && item.path ==
  drag.hoverTarget`) and the destination pane gets a faint tint. Drop resolution reuses the same
  `computeDropFolder`, so highlight and behavior always agree.
- **`feature/cleanup/CleanupScreen.kt`** — "Large files" + "Duplicate files" wrapped in a
  `CollapsibleSectionHeader` (chevron), **collapsed by default** (`mutableStateOf(false)`), headers +
  counts always visible; rows render Coil `AsyncImage` thumbnails (icon fallback via
  `rememberVectorPainter`); the `ReclaimBar` gets `navigationBarsPadding()` so it clears the system
  navigation buttons.
- **`data/index/IndexingScheduler.kt`** — `ensureIndexed()` enqueues the survey with
  `ExistingWorkPolicy.KEEP` (never interrupts an in-progress run), distinct from the explicit
  `rebuildNow()` (REPLACE).
- **`JupiterApp.kt`** — on start, a guarded IO coroutine checks `fileIndexRepository.stats().first()`
  and calls `indexingScheduler.ensureIndexed()` **only when `indexedCount == 0`** — the initial
  thorough survey builds once in the background; the existing real-time delta hooks + `DownloadIndex­Observer`
  keep it live thereafter. Kept `@HiltAndroidApp`, `Configuration.Provider`, `ImageLoaderFactory`, and
  the observer start.

## Lessons captured this iteration

- **A drag drop-target must be resolved with the SAME function used for the highlight** (`computeDropFolder`),
  updated inside `onDrag` — otherwise the tint and the actual drop can disagree.
- **Edge-to-edge (`enableEdgeToEdge`) means bottom bars need explicit `navigationBarsPadding()`** or they
  render under the system nav buttons.
- **Auto-index gating**: enqueue the initial survey only when `indexedCount==0` with `KEEP` so it builds
  once and never re-walks or interrupts itself; deltas do the rest. (Sync to
  `…/lessons-system/lessons_orchestrator.md`.)
