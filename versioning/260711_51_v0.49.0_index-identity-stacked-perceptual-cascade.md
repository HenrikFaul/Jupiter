**Date:** 2026-07-11
**Change package:** 260711_51
**Version:** 0.49.0
**Scope:** Jupiter Android app development (not managed device files).

## Context

A 15-point indexing/dedup architecture spec was (re)submitted. A full code audit was run FIRST so
this round fixes what is genuinely missing without repeating or regressing what earlier versions
already delivered. The audit's verdict per point, and what this round changes, is the authoritative
status below — **the indexing architecture is NOT claimed complete.**

## Key fixes in detail

1. **Identity normalization (11).** All identity comparisons are second-precision
   (`oldMtime/1000 == newMtime/1000`), including the `hashIfUnchanged` SQL. Preservation now also
   covers the new `phash`/`ahash`/`quickHash` columns across upsert/replaceChildren/indexFile/rename.
2. **Stacked perceptual layer (5.3).** `PerceptualHash` gains `aHashFromLuminanceGrid` (8×8 mean),
   `pHashFromLuminanceGrid` (32×32 → DCT-II → 8×8 block, DC excluded, median threshold),
   `combinedScore` (w=0.35/0.45/0.20) with STRICT=5.0 / RELAXED=10.0 tiers, and `isVisuallySimilar`
   (stacked verdict with dHash-only fallback). `BitmapPerceptual` + `PerceptualHashSource.computeAll`
   produce all three layers from ONE decode; the backfill worker and the arrival detector store the
   full fingerprint; the work-list query upgrades legacy dHash-only rows. The LSH near-merge in
   `nearDuplicateImageGroups` now verifies candidates with the stack.
3. **Quick-hash cascade (5.2).** `duplicateGroups` splits same-size candidates by a persisted
   SHA-1(head 64 KiB + tail 64 KiB) before full hashing; full hash remains the only grouping
   authority.
4. **Single rescan (9)** and **real disable (8)** as summarized in the table.

## Verification (real, in CI)

- **`PerceptualStackTest`** (pure JVM): aHash/pHash determinism, small-perturbation tolerance vs
  structural separation, the combined-score formula and tiers, identical-dHash-but-far-pHash/aHash →
  NOT similar (no single hash decides), legacy fallback, UNHASHABLE never matches.
- **`IdentityPrecisionTest`** (Robolectric + real Room): sub-second mtime rounding preserves all four
  hashes and re-stamps the generation; a ≥1 s modification clears them; `hashIfUnchanged` matches
  across precisions.
- **`QuickHashCascadeTest`** (Robolectric + real files + real Room): identical files group; same-size
  files differing only in the MIDDLE (quick hashes collide) are split by the strong hash — proving
  the quick stage is a pre-filter, never the authority; quickHash persists.
- Existing suites unchanged and expected green (the dHash fallback makes the stack non-regressive).

## Remaining risk / rollback

- Moderate. The migration is additive (ALTER TABLE + CREATE INDEX; no data rewrite). Legacy rows are
  upgraded lazily by the backfill (one more pass over the library — foreground, minutes). The
  stacked verification only *tightens* near-merges for fully-fingerprinted pairs; exact groups are
  unaffected. Rollback = revert (a downgraded APK would destructively recreate v4, as before).

## Lessons captured this iteration

- **Normalize identity at every comparison, not at one writer.** Two enumerators with different
  timestamp precision silently disagree about "unchanged" — and the cost was invisible (hashes just
  kept vanishing and being recomputed).
- **A cache with expensive derived data deserves a real migration.** "It's just a cache, wipe it" was
  true until fingerprinting a 39k-photo library became a minutes-long foreground job.
- **Audit before implementing a re-sent spec.** Most of its points were already shipped; the audit
  found the two real bugs hiding between the done items.
