# Jupiter v0.52.0 — index freshness and structural duplicate coverage

- **Version:** `0.52.0` (`versionCode` 4)
- **Date:** 2026-07-12
- **Change package:** 260712_54
- **Scope:** existing Jupiter Android app; no new app, no package rename.

## Goal / why

The user asked to keep the application name as **Jupiter**, then close the remaining functional gaps from the attached requirement extract: continuous background indexing after first launch, immediate useful data on app open, and stronger duplicate handling for files and images whose names, formats, or sizes differ.

## Added

- `app/src/main/java/com/jupiter/filemanager/data/index/StructuralFingerprintBackfillWorker.kt` — WorkManager/Hilt worker that proactively fills text/code, archive/APK, video, PDF, and audio structural/media fingerprints for existing indexed files.
- `FileIndexRepository.nearDuplicateStructuralGroups()` and `FileIndexRepositoryImpl.nearDuplicateStructuralGroups()` — general Similar-tab grouping for non-image near duplicates.
- `app/src/test/java/com/jupiter/filemanager/data/index/NearDuplicateStructuralGroupTest.kt` — Room/Robolectric proof that text, archive, and video structural groups surface as `similar = true` review groups.

## Modified

- `DownloadIndexObserver` — MediaStore changes now kick both `ensureIndexed()` and dedup reconcile.
- `IndexingScheduler` — periodic freshness now runs every 15 minutes with battery-not-low; structural backfill can be scheduled/cancelled with the rest of the pipeline.
- `IndexingWorker` — after a successful full survey, exact duplicate candidate hashes are warmed, then both image and structural/media backfills are chained.
- `JupiterApp`, `MainActivity`, `SettingsViewModel` — startup, foreground, and enable-indexing paths schedule the complete freshness/backfill set.
- `DuplicatesViewModel` — Similar presentation merges image perceptual groups with non-image structural/media groups.
- `README.md`, `changelog.md`, `app/build.gradle.kts` — documented behavior and version bump.
- `C:/Work/Henris/HenrisForge/lessons-system/codingLessonsLearnt.md` — new Android lesson about observer-triggered freshness versus dedup-only catch-up.

## Remaining risk / rollback

- Android OEM Doze policies can delay WorkManager periodic work; the app therefore combines live observer, foreground catch-up, and periodic refresh instead of promising impossible always-on closed-process wakeups.
- Media/PDF/audio fingerprints are conservative descriptors. Undecodable files are marked `UNHASHABLE` and never auto-cleaned.
- Rollback: revert this package; the existing exact byte duplicate and image perceptual layers continue to work.
