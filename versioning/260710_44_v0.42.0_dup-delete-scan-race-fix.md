# v0.42.0 — Deleted duplicate groups no longer reappear (scan-vs-delete race fixed)

**Date:** 2026-07-10
**Change package:** 260710_44
**Version:** 0.42.0
**Scope:** Jupiter Android app development (not managed device files).

## Reported bug (user, with screenshots)

Selecting duplicates and deleting them leaves the deleted sections in the list — they do not
disappear. Deleting every redundant copy in a group (keeping one) should immediately drop the whole
group (it is no longer a duplicate), but the groups stay.

## Root cause

A race between the **silent background rescan** and the delete. The screenshot shows **"Still
scanning…"**: on open, the ViewModel renders the cached list and kicks a silent rescan
(`scan(silent = true)`). That scan's `onCompletion` did `_uiState = copy(groups = result)` — and the
`result` was collected from the index BEFORE the user's delete (a long scan over hundreds of groups).
So the moment the scan finished it **overwrote the post-delete list**, re-adding the groups the user
had just removed. The delete itself worked (files went to the Recycle Bin); the scan clobbered the
UI state afterwards.

## Fix (feature/cleanup)

- The ViewModel now keeps a session `deletedPaths` set. `deleteSelected` adds every successfully
  trashed path to it.
- A single `publishGroups()` funnel sets `groups`, filtering out `deletedPaths` (per file) and
  dropping any group left with fewer than two copies. Scan-collect (streaming), scan-completion, and
  the delete all publish through this rule, so a scan finishing AFTER a delete can never resurrect a
  removed group.
- `deletedPaths` is cleared on a user-initiated fresh scan (Refresh / permission-resume / first
  scan): the index no longer contains the trashed files (their rows were removed by `moveToTrash`),
  so the fresh result is already correct and the session filter can reset.

## Verification

- Compile + unit tests in CI. The fix is ViewModel state-ordering under a real scan/delete race,
  confirmed on a device: with a background scan running ("Still scanning…"), select and delete a
  group's extra copies → the group disappears immediately and stays gone when the scan completes.
  No unit test is added because reproducing the race requires the live index scan flow + trash IO the
  emulator-less harness does not exercise; the fix is a small, self-evident single-funnel invariant.

## Remaining risk / rollback

- Low. `publishGroups` centralises the filter; `deletedPaths` is per-session and reset on a fresh
  scan, so a restored file re-appears after a Refresh. Rollback = revert.

## Lessons captured this iteration

- **Two writers to one UI list must reconcile, not race.** A background refresh and a user mutation
  both writing `groups` will clobber each other unless every write goes through one funnel that
  applies the pending user mutations. The silent-refresh UX (great for "instant open") introduced
  exactly this second writer; the delete-filter funnel makes the two orderings commute.
