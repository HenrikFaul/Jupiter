# v0.22.0 — Index-backed analytics: no deep scan on every open

**Date:** 2026-07-05
**Change package:** 260705_27
**Version:** 0.22.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

Re-stated, emphatic user request: like the desktop `ultimate-duplicate-handler`, run a
thorough indexed survey, persist it, keep it live on every file operation, and — critically
— **never re-run a deep scan when the app opens**; once the survey has run, the app should
already "see" every file. Manual scan must still be available.

### Root-cause investigation (grounded)

A full map of the index system (subagent + direct reads) established:
- The persistent index (`file_index` Room table) **is** built by `IndexingWorker` (walks the
  whole tree, stores metadata for every file), auto-runs on first launch (`ensureIndexed`),
  and is kept live by delta hooks on copy/move/delete/restore + `DownloadIndexObserver`.
- **But almost nothing read from it.** `StorageAnalyticsRepositoryImpl` (which powers Cleanup,
  Duplicate cleanup, Storage Analytics, Home) **re-walked `/storage/emulated/0` on every call**
  to `storageOverview` / `findLargeFiles` / `findDuplicates`. The index was used only as a
  hash-cache inside duplicate detection — the expensive tree walk was never skipped. That walk
  IS the "deep scan on every open" the user experienced.
- Two delta hooks were also missing: `FileRepositoryImpl.rename` and `.createFolder` never
  touched the index, so a renamed file's old path lingered until the next full rebuild.

## Added / Modified

- **`data/index/FileIndexDao.kt`** — new queries: `fileCount()` (populated probe), `largeFiles(minSize, limit)`,
  `allFiles()` (overview aggregation), `collidingSizes(minSize)` + `filesOfSize(size)` (duplicate
  candidate buckets, computed in SQL — no walk).
- **`domain/repository/FileIndexRepository.kt` (+`…/data/index/FileIndexRepositoryImpl.kt`)** —
  new: `isPopulated()`, `largeFiles(minSizeBytes, limit)`, `allFiles()`,
  `duplicateGroups(minSizeBytes)` (size-collision buckets from the DB, confirmed by content hash
  with cache reuse, no filesystem walk), and `hashCollidingSizes(minSizeBytes)` (precompute dup
  hashes). Shared private `hashForEntry` (stored hash → cached-unchanged hash → compute+cache).
- **`data/index/IndexingWorker.kt`** — second survey phase: after the metadata pass, calls
  `hashCollidingSizes(MIN_DUPLICATE_SIZE)` (best-effort, cancellable) so duplicate groups are
  precomputed and appear instantly in Cleanup after the survey.
- **`domain/repository/StorageAnalyticsRepository.kt` (+impl)** — `storageOverview` /
  `findLargeFiles` / `findDuplicates` gained a `preferIndex: Boolean = true` flag. When true and
  the index is populated, results are served instantly from the index (via new helpers
  `indexIsPopulated()` + `overviewFromIndex(volume)` which reuse the exact `categoryForPath` /
  `isExcludedPath` logic of the live walk). On any index failure or an empty index they fall
  through to the original live walk — no regression. `observeStorageOverview()` emits one instant
  index overview when populated, else the progressive walk.
- **`data/file/FileRepositoryImpl.kt`** — injected `FileIndexRepository`; `createFolder` now
  `indexFile`s the new folder and `rename` now `onMovedOrRenamed`s (both best-effort), closing
  the stale-index gaps.
- **`feature/cleanup/CleanupViewModel.kt` + `CleanupScreen.kt`** — `scan()` is index-first
  (instant when populated); new `rescan()` (the ⟳ action) forces a fresh walk AND kicks
  `IndexingScheduler.rebuildNow()`. UI state gained `fromIndex` + `indexedCount`; a new
  `IndexStatusCard` banner ("Instant results from your file index — N indexed …" + Rescan)
  makes the index visibly in use.

## Acceptance criteria (met)

- After the survey has run, opening Cleanup / Duplicate cleanup / Analytics / Home shows results
  **from the index, instantly, with no filesystem walk**. ✔
- Duplicate groups are precomputed during the survey (size-collision hashing) → instant. ✔
- New downloads / edits keep the index live (existing deltas + observer; rename/createFolder gaps
  now closed). ✔
- A manual full rescan is available (Cleanup ⟳ → fresh walk + index rebuild). ✔
- Empty index → automatic fallback to the live walk (first-launch path unchanged). ✔

## Regression checks (verified)

- All existing callers compile unchanged via interface default args (`preferIndex = true`):
  Home `storageOverview()`, Analytics `observeStorageOverview()`, Duplicates `findDuplicates(root)`.
- Only concrete implementors are the real impls (no test fakes broke). DI binding for
  `FileIndexRepository` exists (`IndexModule`); no dependency cycle (index repo depends only on
  DAO+dispatcher).
- Index path reuses the SAME category classification and exclusion filter as the walk, so the
  overview is equivalent. Duplicate confirmation still requires a real content-hash match.
- Brace/paren balance verified on all 9 edited files.

## Remaining risk / rollback

- Medium-low. The read path is new but strictly additive with a live-walk fallback on any failure
  or empty index. Duplicate detection still confirms by content hash. Main residual risk: an index
  that is stale because a file changed outside the app with no delta — mitigated by the survey,
  the deltas, and the manual Rescan. Rollback = revert this commit range.

## Lessons captured this iteration

- **Building an index is only half the job — the read side must actually consume it.** Jupiter had
  a fully-built, live-updated index for versions, yet every heavy screen still deep-scanned because
  no consumer queried it. Infrastructure without wired readers delivers none of the user-visible
  benefit. Always trace a feature end-to-end from the UI's data source, not just from the data layer.
- **A cache that only avoids re-hashing but not re-walking still feels like a full scan.** The tree
  walk (syscalls per file) dominated; caching hashes didn't remove it. Identify the actual dominant
  cost before optimizing.
- **Close delta gaps for every mutation, not just the obvious ones** (rename/createFolder were
  missed while copy/move/delete were covered). (Sync to `…/lessons-system/lessons_orchestrator.md`.)
