# Jupiscan v0.59.0 — compact, evidence-rich on-device metadata index

- Date: 2026-07-16
- Change package: `260716_00`
- Scope: Jupiscan Android app file-index metadata schema, persistence, migration and duplicate comparison

## Goal / authoritative requirement

The user asked to continue creating and saving file metadata so it supports indexing and duplicate detection as strongly as possible while the metadata database occupies minimal phone storage. Functional regression is forbidden.

The selected design enriches the existing single Room index instead of introducing a second catalogue. Metadata already computed during hash/decode work is packed into fixed-width binary/integer fields, persisted lazily, versioned, and invalidated with the byte identity from which it was derived.

## Evaluated concepts

1. **Normalized descriptor side tables with one row per hash/sample.** Flexible, but every row adds SQLite record, primary-key, foreign-key and B-tree overhead. It also makes a five-frame video descriptor several rows and increases join/read cost.
2. **Compact nullable columns on the existing file row.** A 20-byte digest, bounded raw long vector and packed geometry add no extra row/key overhead and preserve current DAO ownership. This was selected as the smaller, lower-regression design.

Full cryptographic evidence is not truncated. Compression is representation-only: hexadecimal text becomes its original raw bytes.

## Schema and storage contract

| Evidence | Legacy representation | v10 representation | Payload effect |
|---|---:|---:|---:|
| Full SHA-1 | 40 ASCII bytes | 20-byte BLOB | 50% of prior payload |
| Quick SHA-1 | decimal size + `:` + 40 ASCII bytes | 20-byte BLOB; size reuses `sizeBytes` | removes duplicate size text and halves digest |
| 5-sample media vector | five 16-hex longs + four commas = 84 ASCII bytes | 40-byte BLOB | 47.6% of prior payload |
| Width + height | absent/separate values would require extra fields/text | one packed 8-byte INTEGER | one nullable scalar |

SQLite record/page overhead is not included in these payload figures. No universal whole-database shrink percentage is claimed.

### Migration invariants

1. Additive nullable columns are created first.
2. A valid 40-hex SHA-1 is converted losslessly to 20 raw bytes and its duplicate TEXT is cleared.
3. Valid legacy quick hashes reuse the existing `sizeBytes` and store only raw digest bytes.
4. Valid v2 ordered media signatures become raw 64-bit samples.
5. Invalid or non-standard legacy tokens remain untouched in TEXT and remain queryable.
6. Only after data conversion is the retired TEXT hash index dropped and the BLOB lookup index created.
7. No destructive fallback and no user-file rescan are used.
8. `VACUUM` runs later, outside the Room migration transaction, through retryable WorkManager maintenance.

## Metadata lifecycle and comparison invariants

- Byte-derived full/quick evidence is reusable only while size and second-normalized mtime identity is unchanged.
- Type-specific image/media descriptors additionally require unchanged `FileType`.
- When current bytes differ, storing the fresh content hash atomically clears every derived quick/perceptual/structural/media/geometry field.
- Exact duplicate identity remains full content hash only.
- Geometry is a conservative, orientation-independent 8% aspect-ratio veto for image/video Similar comparison; missing rolling-backfill geometry remains compatible.
- Media vectors remain ordered and algorithm-versioned; malformed or version-mismatched evidence fails closed.

## Acceptance criteria

- Production SHA-1 writes use the 20-byte BLOB and exact lookups use its SQLite index.
- Existing valid v9 metadata migrates without opening or hashing a user file.
- Unknown legacy tokens are not discarded.
- Compact media vectors roundtrip with order and signed 64-bit values intact.
- Same-byte rename/type discovery retains exact proof, but a type change cannot reuse the old type descriptor.
- A byte/mtime change cannot expose a stale descriptor under a fresh content hash.
- Resized/rotated same-shape media can match; clearly incompatible aspect ratios cannot.
- All previous duplicate/index unit tests, lint and debug assembly remain green.
- App launch on API 34 produces schema v10 and completes versioned physical compaction without a crash.

## Post-delivery VISOR specification boundary

The subsequently attached Jupiscan-VISOR brief was reviewed. Its algorithms are useful roadmap input, but its integration contract explicitly assumes a Python desktop/PyQt-or-Tauri plugin tree, Python dependencies, FAISS files and a separate feature database. This repository is the existing Kotlin/Jetpack Compose Android application. Creating the specified Python subsystem here would violate the existing-app and single-index constraints and would not be an Android deployment.

This v0.59.0 release therefore claims only the native compact metadata, stacked image hash, type-aware media vector and geometry work proven above. A future VISOR port must be separately specified as an Android-native, mobile-budgeted pipeline (or an explicitly authorized external service), with benchmark/calibration data and a schema plan before adding heavier descriptors. No nine-layer VISOR completion claim is made in this release.

## Remaining risk / rollback

- On a very large existing database, WorkManager may defer VACUUM until battery is not low; the logical schema remains usable meanwhile and retry is safe.
- Physical file shrink depends on page occupancy and legacy metadata coverage. Measure on the target Samsung/40k corpus before publishing a percentage claim.
- Precision-first geometry can create a false negative for deliberately stretched/cropped media. It cannot promote anything to exact and is rollbackable by disabling the geometry veto while retaining schema v10.
- Do not downgrade an installed v10 database to a v9 APK. Forward rollback must keep schema v10 and dual-read fields, then disable only new producers/comparison gates.

## Lessons captured

`LESSON-ANDROID-COMPACT-METADATA-LIFECYCLE-026` is appended to the external HenrisForge Android section: representation compaction must be lossless, physical SQLite shrink must be deferred outside migration, and derived evidence must share the source identity lifecycle.
