# v0.30.0 — Granting All Files Access reliably triggers the full-storage survey

**Date:** 2026-07-06
**Change package:** 260706_32
**Version:** 0.30.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

User report (verbatim intent): *"folytasd, hogy ha hozzáférést adok akkor a teljes tárhely
scanneléséhez kapjon hozzáférést az app"* — "if I GRANT access, the app should GET access to scan
the FULL storage." The frustration was that after granting All Files Access nothing appeared to
scan. Two things were needed: (a) make the grant reliably kick off a full-storage survey without a
cold restart, and (b) be honest, in-app, about the ~190 GB of app-private data that NO file manager
can browse on Android 11+ (a platform limit, not a Jupiter bug).

## Root cause

The first-run survey is scheduled at process start (`JupiterApp.onCreate` → `IndexingScheduler
.ensureIndexed()`), which runs **before** the user has granted All Files Access. `IndexingWorker
.doWork()` gates on `hasAllFilesAccess()` and, when it's false, **no-ops** (`Result.success(0,0)`
without `beginScan()`), leaving `index_state` at `EMPTY`. Nothing re-triggered a survey once access
was finally granted — the only re-trigger was the *next* cold start. So "I gave access but it still
doesn't scan" was accurate: the grant took effect only after a full app restart.

## Fixed this iteration

- **Grant → scan trigger.** `PermissionViewModel.refresh()` (already called on every `ON_RESUME`
  and after the legacy-permission launcher) now, when access is present, starts the full-storage
  survey. The decision is a pure, unit-tested policy — `GrantSurveyPolicy.shouldStartSurvey(
  hasFullAccess, indexingEnabled, status)` — which returns true only when access is held, indexing
  is enabled, and the index is neither `RUNNING` nor `COMPLETE` (i.e. `EMPTY`/`DIRTY`/`FAILED`). It
  enqueues via `IndexingScheduler.rebuildNow()` (REPLACE), which cancels the earlier no-op work and
  starts a fresh pass that can actually see the volume.
- **Race hardening (from adversarial review).** The scheduling coroutine runs on a new
  process-lifetime `@ApplicationScope` `CoroutineScope`, **not** `viewModelScope`. Granting access
  navigates away with `popUpTo(Permission, inclusive = true)`, which clears the ViewModel and would
  cancel a `viewModelScope` job mid-flight (before the DataStore/Room reads finish and the enqueue
  lands) — most visibly on devices with system animations disabled, where the clear is immediate.
  On `appScope` the enqueue is guaranteed to run.
- **Honesty / factual accuracy (confirmed review finding).** The two new user-facing texts
  (permission screen note + App-storage explainer card) were corrected so they no longer attribute
  installed-APK inaccessibility to Android 11 scoped storage: `Android/data`/`Android/obb` are the
  Android-11 scoped-storage restriction (off-limits even with All Files Access), whereas installed
  APKs (`/data/app`) and app caches were never browsable by a non-root file manager on any version.

## Verification (real, in CI)

- **`GrantSurveyPolicyTest` (pure JVM, `testDebugUnitTest`)** — exhaustive truth table proving the
  user-facing contract:
  1. access on + indexing on + `EMPTY` → **starts** the survey (the bug's fix);
  2. `FAILED`/`DIRTY` → retries;
  3. `RUNNING` → never restarts an in-flight scan;
  4. `COMPLETE` → never redundantly rescans;
  5. no full access → never starts (all 5 statuses);
  6. indexing disabled → never starts (all 5 statuses), so the opt-out is respected.
- Hilt graph checked: `IndexStateRepository` is `@Binds`-bound (`IndexModule`); `IndexingScheduler`
  and `SettingsDataStore` are `@Inject @Singleton`; the new `@ApplicationScope CoroutineScope` is
  `@Provides @Singleton` in `CoroutineModule`.
- **Adversarial review** (4 lenses — correctness/lifecycle, DI/compile, WorkManager semantics,
  regression/honesty — each finding skeptic-verified): 1 confirmed (the APK-attribution wording,
  fixed above) + 1 refuted-but-hardened (the `viewModelScope` teardown race, eliminated via
  `appScope`).
- **CI (authoritative):** to be confirmed green (compile + unit + Robolectric) after push.

## Remaining risk / rollback

- Low. The change is additive: a new pure policy, a new DI-provided app scope, one ViewModel trigger,
  and two corrected text strings. The trigger is idempotent (policy gates on `RUNNING`/`COMPLETE`) and
  fire-and-forget (`runCatching`), so it can never crash the permission flow. Rollback = revert this
  commit; the prior grant-on-next-cold-start behavior returns.

## Lessons captured this iteration

- **A permission grant is a lifecycle edge — wire the side effect to it.** Scheduling the survey only
  at process start meant the grant (which happens later, in a different process moment) had no effect
  until the next launch. The fix is to react to the access-state transition, not just to startup.
- **Fire-and-forget that must survive a screen must not ride `viewModelScope`.** Navigation that pops
  the screen cancels its ViewModel scope; work that has to outlive the screen belongs on a
  process-lifetime scope.
- **"Honest" copy has to be technically honest.** App-private inaccessibility has two distinct causes
  (scoped storage for `Android/data`; Linux/SELinux for APKs) — conflating them under "Android 11"
  is a real inaccuracy. (Sync to `…/lessons-system/lessons_orchestrator.md`.)
