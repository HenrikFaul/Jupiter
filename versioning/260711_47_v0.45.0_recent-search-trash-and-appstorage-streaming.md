# v0.45.0 — Recent/search `.Trash` "Not found" closed; App-storage scan streams (instant "Scanning…")

**Date:** 2026-07-11
**Change package:** 260711_47
**Version:** 0.45.0
**Scope:** Jupiter Android app development (not managed device files).

## Reported items (user, with screenshots)

1. **Still "Not found" on a Samsung `.Trash` file.** Screenshot: Preview → red "!" →
   `Not found: /storage/emulated/0/Android/.Trash/com.sec.android.app.myfiles/…/Download/…/mediadownload (9).apk`.
   The user demanded a thorough, technological root-cause fix — "this is clearly a deleted-file bug".
2. **App storage half-fixed.** After granting Usage-access the scan now starts on its own, but the
   "Grant Usage access" text stays for **9–15 s** (measured across ~7 runs) before the info appears —
   unreasonably slow and misleading. Requested: show **"Scanning in process" immediately**, and show
   at least the **first ~5 scanned apps at once**, then keep filling in the rest as the scan proceeds,
   so the user can see the app is working.

## Root cause (thorough investigation)

A full-repo inventory of every file-list surface (and every route to the Preview screen) found the
`.Trash` leak came from the **one remaining enumerator that never applied `StorageExclusions`**: the
filesystem-walk search, `FileRepositoryImpl.search()` (`walkTopDown` + `matchesFilter`, which only
checks hidden/name/type). It backs **two** user surfaces:
- **Recent tab** — `RecentViewModel.scanStorage()` calls `fileRepository.search(root, NO_FILTER)` and
  keeps the 60 most-recently-modified files. A just-trashed `.apk` has a fresh mtime, so it lands at
  the TOP of Recent every scan. **This is the "keeps appearing" surface.**
- **Global search walk fallback** — `SearchViewModel` when the index isn't COMPLETE / NL query.

The trashed `.apk` is **not** dot-prefixed, so `showHidden=false` never hid it. Opening it → Preview →
`getFile()` → `NotFound` (the name carries bytes that don't round-trip through Java path encoding, and
Samsung purges the bin, so `File.exists()` is false). Every OTHER surface already excluded it
(category/album/index-read/analytics/index-build — v0.37→v0.44).

A second, latent leak: `StorageAnalyticsRepositoryImpl.findLargeFiles()`'s live-walk branch omitted
the `isExcludedPath` guard its own siblings (`findDuplicates`, `storageOverview`) all apply.

The App-storage slowness: `AppStorageSource.query()` walked all ~740 packages and returned the whole
overview only at the end; meanwhile `AppStorageViewModel.load()` kept `permissionRequired=true` in
state, so the screen showed the grant prompt for the entire 9–15 s scan, then flipped to the full list.

## Fixes / changes

- **(1) `FileRepositoryImpl.search()` excludes trash/thumbnail/app-private paths** — one `continue` on
  `StorageExclusions.isExcluded(file.absolutePath)` in the walk loop, closing the Recent tab AND global
  search fallback in a single boundary. (data/file)
- **(1b) `StorageAnalyticsRepositoryImpl.findLargeFiles()`** live-walk branch now calls
  `isExcludedPath(safePath(file))` like its siblings — the Cleanup Large-files list never surfaces a
  trash row even when the index isn't usable. (data/storage)
- **(2) Streaming App-storage scan.** New `AppStorageSource.queryStream(): Flow<AppStorageOverview>`:
  emits a `permissionRequired` frame if access is missing; otherwise an immediate empty `scanning=true`
  frame (drops the grant prompt AT ONCE → "Scanning app storage…" view), then a size-sorted partial
  snapshot after the first 5 apps and every 25 thereafter, then a final `scanning=false` complete
  result. `AppStorageOverview` gains a `scanning` flag. `AppStorageViewModel.load()` clears
  `permissionRequired` SYNCHRONOUSLY when access is already held (so the prompt never lingers) and
  collects the stream; a refresh keeps the existing list visible with a "Scanning… N apps so far"
  banner instead of blanking. The screen shows `ScanningView` before the first apps arrive and a
  compact `ScanningBanner` while partial results are shown. The unused single-shot `query()` was
  removed (no divergent second copy of the scan loop). (data/apps + feature/apps + domain/model)

## Remaining risk / rollback

- Low. `search()` gains one predicate call per walked entry (cheap, already used by the walk's other
  consumers). `findLargeFiles` now matches its siblings. The streaming scan is additive; the ViewModel
  cancels the prior scan job on each load (no overlap). Rollback = revert.

## Lessons captured this iteration

- **"Cover every read" means every enumerator, including the filesystem-walk one.** v0.43/v0.44 routed
  MediaStore, index reads and analytics through the one exclusion, but the raw filesystem-walk search
  — which powers Recent (a permanent tab) — was the last unguarded boundary, so the same "Not found"
  kept resurfacing there. A cross-cutting rule is only as strong as its least-guarded call site.
- **A grant is not the same as a result.** Detecting access and showing the answer are two events; a
  long synchronous scan between them must transition the UI to a "working…" state on the FIRST event,
  and stream partial results, or the user reads "still not granted" for the whole scan.
