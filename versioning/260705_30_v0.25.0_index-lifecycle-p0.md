# v0.25.0 — Index life-cycle P0 fixes (expert architecture review)

**Date:** 2026-07-05
**Change package:** 260705_30
**Version:** 0.25.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

A detailed expert review of the indexing system identified that the real problem is a
**broken index life-cycle**, not a single bug: any non-empty Room table was treated as a
finished index; full rebuilds nulled cached hashes; there was no global stale sweep; directory
copy/move/rename/restore only updated the top-level entry; and — most visibly — the search
screen deliberately re-walked the whole volume after showing index results. The review supplied
a P0/P1/P2 ordering and an acceptance-test suite, and explicitly said not to accept "it compiles"
as proof.

This iteration lands the highest-impact P0 fixes that are correct and unit-testable now, and is
transparent about what needs the larger generation/schema redesign and on-device tests.

## Fixed this iteration (P0 subset)

- **#1 — completeness, not row count.** New `SettingsDataStore.indexComplete` flag (DataStore).
  `IndexingWorker.doWork` sets it `false` at start and `true` only after the full survey succeeds;
  a cancelled/killed run leaves it `false`. `JupiterApp` now (re)builds when `!indexComplete`
  instead of when `indexedCount == 0`, so a partial/interrupted index is never treated as done and
  rebuilds on next launch. (`data/preferences/SettingsDataStore.kt`, `data/index/IndexingWorker.kt`,
  `JupiterApp.kt`.)
- **#3 — search stops re-walking storage.** `SearchViewModel.search()` now serves a plain-text
  query from Room ONLY (`indexRepository.search`) once `indexComplete` is true — zero
  `walkTopDown`. The live-walk path remains only as a fallback until the first full index exists,
  or for natural-language queries. (`feature/search/SearchViewModel.kt`.)
- **#4 — rebuilds preserve hashes.** `FileIndexRepositoryImpl.upsert` now batches a
  `entriesForPaths` lookup and preserves a stored `contentHash` when the file identity (size+mtime)
  is unchanged, instead of nulling every hash via `toEntry`. (`data/index/FileIndexRepositoryImpl.kt`,
  `data/index/FileIndexDao.kt`.)
- **#5 — directory rename/move preserves the whole subtree.** `onMovedOrRenamed` now rewrites every
  descendant row's path/parentPath under the new root (preserving each cached hash) via a new
  `dao.entriesUnder` + the pure `IndexPathRewrite`, instead of dropping the subtree and re-indexing
  only the new root (which lost every descendant until the next full scan).

## Added

- **`data/index/IndexPathRewrite.kt`** — pure, dependency-free path arithmetic (`rewrite`,
  `parentOf`, `nameOf`, `identityUnchanged`), segment-exact so a sibling sharing a textual prefix
  (`Photos` vs `Photos_2024`) is never rewritten.
- **`app/src/test/.../IndexPathRewriteTest.kt`** — JVM unit tests (run in CI `testDebugUnitTest`)
  covering root move, descendant rewrite, sibling-prefix safety, trailing slashes, and the
  hash-preservation predicate.

### Honest testing limitation

The review's acceptance suite (interrupted-scan-not-complete, second-launch-no-walk, external
delete/rename reconciliation, 100-file directory copy, etc.) is **instrumentation/device testing**.
This CI runs only JVM `testDebugUnitTest` (no emulator/Robolectric), so those end-to-end tests are
NOT claimed to pass here. Enabling them requires adding Robolectric (for Room DAO JVM tests) and/or
an emulator step — planned for the next round. This record does not assert those tests ran.

## Remaining risk / rollback

- Medium. `onMovedOrRenamed`'s delete-then-insert is not a single Room transaction (best-effort
  index; a crash mid-op self-heals on next scan). Index-only search may miss very recently-added
  files until the index catches up (intended per requirement #10). Rollback = revert this commit.

## Lessons captured this iteration

- **Completeness is a state, not a count.** "Rows exist" ≠ "index is done"; track scan completion
  explicitly or a killed build masquerades as finished.
- **A rename must move the subtree, not delete it.** Reflect structural ops by rewriting descendant
  paths (preserving hashes), never drop-and-reindex-the-root.
- **Be honest about proof.** Unit-test the pure logic in CI; do not claim device acceptance tests
  passed when the CI cannot run them. (Sync to `…/lessons-system/lessons_orchestrator.md`.)
