# Jupiter v0.50.0 — design rendszer, biztonságos munkafolyamatok és valós UI

- **Dátum:** 2026-07-12
- **Change-package id:** `260712_52`
- **Verzió:** `0.50.0` (`versionCode` 2)
- **Scope:** kizárólag a Jupiter Android alkalmazás; nem a felhasználó készülékén kezelt fájlok.

## Cél / miért

Az authoritative kérés szerint a már létező Android file managerre kellett „**teljes mértékben ráhúz[ni] a promptok alapján a mellékelt designeokat és képeket**”, a meglévő funkciók és direkt elnevezések regressziója nélkül. A designban vagy promptokban szereplő, de hiányzó működést is meg kellett valósítani, a `versioning`, `changelog`, Henris lessons és FORGE útmutatók figyelembevételével.

Ez a kör ezért nem új alkalmazást épít: a valós indexelési/deduplikációs, fájlműveleti és tárolókezelési rétegekre alkalmaz egy közös sötét–türkiz vizuális nyelvet, megszünteti a felismert funkcionális zsákutcákat, és külön adatbiztonsági regresszióvédelmet tesz az átviteli útvonalra.

## Kontroll, prompt és governance

- Egy fájlt egyszerre egy felelős ág módosított; a párhuzamos feladatok a funkcionális audit, Vault import, keresés, app storage, kategória/fotó, Settings és README scope-jaira voltak szétválasztva.
- A FORGE UI-elv szerint a látvány kizárólag valós state-re épül: nincs fiktív tárhelyszázalék, fake „Similar” szűrő, fake PIN/language/auto-lock beállítás vagy gyártott keresési kivonat.
- Az index/dedup v0.49 autoritatív state, exact-vs-similar, quality-keep, Recycle Bin és `StorageExclusions` invariánsait megőriztük. Az egyetlen index-módosítás a közös kizárási policy használata, ami éppen a stale Kuka-sorok felkínálását zárja ki.
- Az új beállítások és UI-state mezők alapértékkel indulnak; publikus screen/route/enum nem lett eltávolítva vagy átnevezve.

## Regressziós ellenőrzések

- **Fájlművelet/adatbiztonság:** a delete továbbra is csak Recycle Binbe visz; a permanent delete továbbra is megerősített; a célütközés preflight az első írás előtt leáll.
- **Index/dedup:** az exact és perceptuális similar csoportok külön maradnak; a UI csak a minőségi sorrend szerinti extra példányokat jelöli; AI-cleanup jelölőt nem vezettünk vissza.
- **Exclusion:** `Android/.Trash`, `.trashed`, `.thumbnails`, `Android/data` és `Android/obb` egységes policy szerint kiesik, míg `content://` és a tokeneket csak névként tartalmazó normál utak maradnak kezelhetők.
- **Navigáció/Hilt:** a debug Kotlin+Java fordítás, KSP és Hilt aggregáció a teljes JVM tesztfeladatban lefutott; a folder open route célzottan Browser.
- **Meglévő scan/remote/vault contractok:** a streaming app-storage scan, index-beállítások, remote implementációk és a régi Vault path API érintetlenül/kompatibilisen maradtak.

## Fennmaradó kockázat / rollback

- A nagy Compose-vizuális változás képernyőméret- és eszköz-specifikus ellenőrzést igényel; rollbackhoz a közös `JupiterDesign` komponensek és az érintett feature képernyők külön visszafordíthatók, az üzleti/index réteghez nem kell nyúlni.
- A dark téma/dynamic color alapérték csak új vagy hiányzó preferenciára hat; felhasználói mentett választás megmarad.
- A `CREATE_NEW` célütközést biztonságosan elutasítja. Ez tudatos viselkedésváltás a korábbi csendes felülírás helyett; ha egy későbbi termékkör felülírást kér, annak külön, explicit megerősítő UX és backup-policy kell.
