# Jupiscan v0.58.0 — precision-first duplicate descriptors and non-transitive groups

- Date: 2026-07-15
- Change package: `260715_00`
- Scope: Jupiscan Android app duplicate indexing/comparison/review pipeline

## Goal / authoritative requirement

The reported real-device result grouped 31 visibly unrelated media files into one duplicate card. The required outcome was to inspect metadata generation and every type-specific comparison algorithm, use the proven desktop implementation in `C:\Work\ultimate-duplicate-handler` as a reference, fix the result without functional regression, and preserve the existing Jupiscan workflows.

The root cause was not the thumbnail UI. Production VIDEO rows contained one middle-frame dHash, accepted up to 10 differing bits, and were union-find clustered. Single-link clustering therefore allowed a weak common frame or an A~B~C bridge to create a heterogeneous component. Image grouping used the same transitive topology, and its exact-dHash first pass could merge before the stacked hashes confirmed the pair. The persistent scan snapshot also discarded `DuplicateGroup.similar`, reconstructing a cached similar group as exact after process restart.

## Reference analysis

`C:\Work\ultimate-duplicate-handler\app\src-tauri\src\engine\videohash.rs` uses an ordered keyframe vector and confirms candidates with full signature distance. Jupiscan ports the invariant—not the desktop dependencies—using Android `MediaMetadataRetriever`, bounded frames, Room persistence, and stricter metadata vetoes. Exact byte identity remains a separate full-content-hash path.

## Control and safety invariants

1. Full content hash is the only exact-duplicate identity.
2. VIDEO requires compatible duration and at least three informative aligned timeline frames; mean/max/80% gates all apply.
3. PDF requires equal page count and agreement across sampled informative pages.
4. AUDIO requires compatible duration and a stricter acoustic-envelope distance.
5. IMAGE requires dHash plus pHash or aHash; no single image hash family decides.
6. Every visible near group is a complete-link partition; candidate graph connectivity alone is insufficient.
7. Comparison-budget exhaustion fails closed to singleton, never blind merge.
8. Descriptor algorithm versions cannot cross-compare.
9. Similar cache state survives restart; unknown legacy snapshot is discarded.
10. Similar output is visibly review-only and keeps the keeper/Recycle Bin safety path.

## Acceptance criteria

- A video sharing one frame with another does not match when the remaining timeline differs.
- Low-information black/flat frames cannot establish similarity.
- A~B and B~C do not produce an A+B+C group when A and C fail direct comparison, for both image and video.
- Duration/page-count mismatch vetoes the corresponding media type.
- A real same-picture PNG→JPEG resize/recompression continues to alert through stacked evidence.
- Old media descriptors are requeued without deleting exact/image/text/archive data.
- A cold cache reload preserves `similar=true`; v1 snapshots cannot become exact groups.
- All prior unit tests and the debug APK build pass.

## Remaining risk / rollback

- Real codec extraction must still be rerun on the reporter's Samsung corpus. JVM tests prove decision logic and Android compilation, not OEM decoder output.
- Precision-first gates can create false negatives for heavily trimmed videos or changed-page-count PDFs; those are safer than false cleanup candidates and may later receive a separate semantic-similarity surface.
- Rollback code only after migrating v9 data consciously; an app downgrade cannot read a newer Room schema. Forward rollback should keep schema v9 and disable the new comparator behind a versioned implementation rather than dropping columns.
