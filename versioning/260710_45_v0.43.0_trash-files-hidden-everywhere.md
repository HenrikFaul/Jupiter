# v0.43.0 — Vendor recycle-bin (Samsung `.Trash`) files no longer surface anywhere → no more "Not found"

**Date:** 2026-07-10
**Change package:** 260710_45
**Version:** 0.43.0
**Scope:** Jupiter Android app development (not managed device files).

## Reported bug (user, with screenshots)

Many files open to a "Not found" error in Preview. The paths are all Samsung My Files recycle-bin
entries: `/storage/emulated/0/Android/.Trash/com.sec.android.app.myfiles/<uuid>/<ts>/storage/emulated/0/Download/…/app-debug (6).apk`.
"Examine how file directories are used/identified and fix ensuring accessibility."

## Root cause

v0.37.0 added the `.trash` exclusion **only to the two index sources** (`MediaStoreIndexSource`,
`IndexingWorker`). But files reach the user through several OTHER enumerators that had **no trash
exclusion**:

- **`MediaStoreCategorySource`** — the category browser (APKs / Downloads / Documents / …). This is
  the screenshot culprit: it lists every APK MediaStore knows, including the ones parked in
  `Android/.Trash/…`, and tapping one opens Preview → "Not found".
- **`AlbumsSource`** — gallery albums; a trash bucket could appear as an album with trashed covers.
- **`FileRepositoryImpl.isIndexableDir`** — the browse-time self-heal exclusion list (missing `.trash`).
- **`StorageAnalyticsRepositoryImpl`** — duplicates / large files / overview (its list was missing
  `.trash` entirely).

Each maintained its own copy of the exclusion list, so the fix in one place didn't cover the others.

## Fix

- **New single source of truth `core/util/StorageExclusions`**: `isExcluded(path)` with full,
  case-insensitive path-SEGMENT matching over `android/data`, `android/obb`, `.thumbnails`,
  `.trashed`, `.trash`. A folder that merely contains the token in its name (`my.trash.notes`) is not
  excluded; content:// URIs (scoped storage) never match, so they stay openable.
- **Applied at every enumeration boundary**: `MediaStoreCategorySource` (skip excluded rows),
  `AlbumsSource` (album folding + images-in-album), `MediaStoreIndexSource`, `IndexingWorker`,
  `StorageAnalyticsRepositoryImpl`, and `FileRepositoryImpl` — all four latter classes had their
  duplicated lists replaced by the shared util.
- **Read-time hiding of stale rows**: `FileIndexRepository.search()` and `observeChildren()` filter
  excluded paths, so trash rows indexed BEFORE the exclusion existed disappear from search/browse
  immediately instead of waiting for the next full survey to sweep them.

## Verification (real, in CI)

- **`StorageExclusionsTest`** (pure JVM): the exact reported Samsung `Android/.Trash/…/app-debug.apk`
  path (case-insensitive), plus `android/data`, `android/obb`, `.thumbnails`, `.trashed` segments are
  excluded; ordinary files, folders with the token only in their name (`my.trash.notes`,
  `androiddata`), and content:// URIs are NOT.
- Existing dedup/index tests unchanged (the index sources' behavior is identical — they already
  excluded `.trash`; this only unifies the implementation).

## Remaining risk / rollback

- Low. The exclusion is centralised and its segment-matching is stricter (fewer accidental
  exclusions) than the previous substring checks in analytics. Content-URI rows are unaffected.
  Rollback = revert.

## Lessons captured this iteration

- **A cross-cutting rule duplicated in N places gets fixed in 1 and stays broken in N-1.** The trash
  exclusion lived in five separate copies; the visible bug was simply the copies that never got the
  v0.37.0 update. Centralising `isExcluded` into one util and pointing every enumerator at it is the
  actual fix — and prevents the next enumerator from re-introducing it.
