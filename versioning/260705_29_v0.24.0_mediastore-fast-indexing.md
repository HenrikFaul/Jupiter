# v0.24.0 — MediaStore-based fast indexing + real progress %

**Date:** 2026-07-05
**Change package:** 260705_29
**Version:** 0.24.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

On-device report (screenshot): the index feature works ("Instant from index · 3570 items"), but the
background SURVEY is pathologically slow — ~40 minutes in, only ~23.5 GB / ~3570 files of 215.7 GB used
was indexed (~10%), and the UI showed only a spinner (no %). User asked to research the best technique
with 10 agents, pick the best, combine, and test.

### Root cause (grounded)

The survey walked `/storage/emulated/0` with `java.io`. On modern Android the shared volume is a
FUSE/sdcardfs passthrough, so every syscall is a slow userspace round-trip, and the current path did
**~6 syscalls per file** (`isDirectory`,`length`,`lastModified`,`isHidden`,`canRead`,`canWrite`) PLUS a
redundant `directory.list()` per directory (childCount) PLUS `file.canonicalPath` (realpath) per
directory. A phase-2 pass also hashed size-colliding media (read gigabytes). The worker was a plain
deferrable `OneTimeWorkRequest` (background-throttled).

### Research (10-agent workflow `wf_bf589356-466`)

Ten agents each investigated one technique (MediaStore bulk, nio single-stat walk, expedited/foreground
execution, parallel traversal, Room-insert tuning, incremental/observer deltas, cheap hashing, progress
estimation, FUSE reality + competitor tactics, native/raw-path). The decisive, unanimous winner:
**MediaStore is Android's own pre-built index of the shared volume** — a single projected
`MediaStore.Files` cursor returns path/size/date/mime for every scanned file with ZERO per-file syscalls
and no walk (~100–1000× for the covered set). This is how Files by Google / Solid Explorer achieve
instant listings.

## Winning combination implemented

- **`data/index/MediaStoreIndexSource.kt` (new)** — device-wide bulk enumeration via a single
  `MediaStore.Files.getContentUri("external")` cursor: `count()` (instant progress denominator) and
  `forEachBatch(batchSize, onBatch)` streaming `FileItem`s in chunks, skipping non-path rows and the
  same excluded segments the survey uses. Never throws.
- **`data/index/IndexingWorker.kt`** — `doWork` is now MediaStore-first: `indexFromMediaStore()` bulk-
  indexes in `BATCH_SIZE` chunks and publishes real `indexed`/`total` progress after each batch; the
  filesystem walk (`indexTree`) remains ONLY as a safety-net fallback when MediaStore returns 0. The
  gigabyte-reading `hashCollidingSizes` phase was removed from the survey (duplicates still hash lazily
  on first Cleanup open via `hashForEntry`, cached).
- **Real % progress** — `KEY_TOTAL_COUNT` added; `SettingsViewModel`/`SettingsUiState` expose
  `indexProgressCurrent/Total/Percent`; `SettingsScreen.IndexStatusRow` shows "Indexing files… N%",
  "current / total files", and a determinate `LinearProgressIndicator` (indeterminate only until the
  first total lands).

## Remaining risk / rollback

- Low-medium. MediaStore is the standard fast path; the walk fallback preserves the old behavior when
  MediaStore is unavailable. Main behavior change: the survey indexes files (not directory rows) and
  omits non-MediaStore files until browsed — an intentional speed/coverage trade the leading file
  managers also make. Rollback = revert this commit.

## Lessons captured this iteration

- **Don't crawl what the OS already indexed.** On Android, MediaStore is a maintained index of the shared
  volume; a `java.io` walk over FUSE re-does that work at ~6 syscalls/file. MediaStore-first is the
  single biggest indexing win.
- **A long job needs a real denominator.** Publishing `indexed/total` (total from an instant count)
  turns an indeterminate spinner into an honest percentage. (Sync to
  `…/lessons-system/lessons_orchestrator.md`.)
