# v0.40.0 — Duplicate cleanup UX: instant re-open, select-all/deselect toggle, nav-bar-safe Delete

**Date:** 2026-07-08
**Change package:** 260708_42
**Version:** 0.40.0
**Scope:** Jupiter Android app development (not managed device files).

## Reported bugs (user, with screenshots of the Duplicate cleanup screen)

1. The ✓✓ toolbar button only ever "selects all" — there is no "deselect all". Make it do both (or add
   a separate button), but the clear action must exist.
2. After the duplicate analysis has run once, re-opening the screen must show the saved state
   INSTANTLY, not a multi-second blank white screen (users close the app after ~3 s).
3. The Delete button is drawn under the system navigation bar, so it can't be tapped — it must be
   responsive/usable.

## Fixes

- **Bug 1 (feature/cleanup):** the ✓✓ action is now a toggle — when nothing is selected it selects
  every extra copy (all but the best in each group); when something is selected it clears the whole
  selection. The icon reflects the state (`DoneAll` ↔ `RemoveDone`) and the content-description flips
  ("Select all extra copies" ↔ "Clear selection"). The existing `clearSelection()` VM action is reused.
- **Bug 2 (feature/cleanup):** new `DuplicateScanCache` (`@Singleton`) holds the last completed
  analysis (groups + probed qualities) for the process lifetime. `DuplicatesViewModel.init` now: if a
  cached result exists, renders it IMMEDIATELY (`isScanning = false`) — an empty cached list correctly
  shows "no duplicates" instantly — and kicks a **silent** background rescan (`scan(silent = true)`)
  that keeps the old list on screen and swaps in the fresh result only on completion (no flicker, no
  blank screen). The cache is also updated after a delete (so a re-open shows the post-deletion list)
  and as quality probes resolve. A non-silent scan now sets the scanning state synchronously so the
  screen never flashes the empty state before the flow starts.
- **Bug 3 (feature/cleanup):** the bottom `DeleteBar`'s content row now applies `navigationBarsPadding()`,
  so the Delete action always sits above the gesture pill / 3-button navigation bar and stays tappable.
  The Scaffold reserves the taller bar's height, so the list is not obscured either.

## Verification

- Logic/compile verified in CI (`assembleDebug`/`testDebugUnitTest`). The three fixes are UI/lifecycle
  behavior (Compose insets, a `@Singleton` in-memory cache, a toolbar toggle) whose end-to-end effect
  is confirmed on a device: re-open the screen → previous results appear with no blank wait; tap ✓✓ →
  selects all, tap again → clears; the Delete bar clears the navigation bar on both gesture and
  3-button navigation. No new unit test is added because the change is inseparable from Compose
  layout + ViewModel lifecycle (recreation across navigation), which the JVM/Robolectric harness does
  not meaningfully exercise; the VM cache seam is simple and its behavior is obvious from the code.

## Remaining risk / rollback

- Low. The cache is additive and in-memory; the silent-refresh path reuses the existing scan flow and
  only changes when results are swapped in. The insets fix is a single modifier. Rollback = revert.

## Lessons captured this iteration

- **A recreated ViewModel needs an external cache to feel instant.** `@HiltViewModel` is rebuilt on
  every navigation to the screen, so "remember the last result" must live in a process-scoped
  `@Singleton`, not a VM field — otherwise every re-open pays the full scan again.
- **Custom bottom bars own their window insets.** Unlike `NavigationBar`, a hand-rolled `Surface`
  bottom bar does not get navigation-bar padding for free; forgetting `navigationBarsPadding()` puts
  its actions under the system bar. Apply it to the content row so the bar's background still extends
  behind the system bar while the controls stay above it.
