## Goal / why

On-device report: within Smart Cleanup the "Large files" and "Duplicate files" sections
should be stacked, **collapsed by default**, and expand only when the user taps the section
name. The user stated the section could NOT be collapsed/expanded by tapping "Large files"
or "Duplicate files".

### Root-cause investigation (grounded, not assumed)

- The collapse LOGIC was already correct and committed in v0.20.0 (`4b0cff8`):
  `largeExpanded` / `duplicatesExpanded` default `false`; file rows are emitted ONLY inside
  `if (largeExpanded)` / `if (duplicatesExpanded)`; the header `onToggle` flips the boolean.
- CI for `4b0cff8` is **green** (android.yml runs #85/#86), so the collapse code is compiled
  into the rolling `build-latest` APK. There is no code path that renders file rows while
  collapsed â€” an up-to-date install shows headers only by default.
- Therefore the "rows always visible, tapping does nothing" screenshots came from a build
  installed **before** #85/#86 published (a timing/stale-install), OR the previous bare-`Row`
  header simply did not read as a tappable control (tiny chevron, no button styling), so the
  user did not realise the whole row was the toggle.

Either way, the durable fix is to make the tap target **unmistakable** and ship a fresh
version so a reinstall is forced.

## Lessons captured this iteration

- **A correct-but-invisible affordance reads as a bug.** The collapse logic was right in
  v0.20.0, but a bare `Row` + small chevron did not look tappable, so the user concluded it
  "doesn't work." When a control's discoverability is in question, style it as an explicit
  control (card/button + verb label) rather than relying on an icon alone.
- **Confirm what the user actually ran.** CI green + committed logic meant the code was fine;
  the gap was install timing / affordance, not the algorithm. Check the published artifact and
  CI status before re-implementing something that already exists. (Sync to
