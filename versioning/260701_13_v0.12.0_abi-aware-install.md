# v0.12.0 — ABI-aware, self-healing install (specific diagnosis + compatible-emulator fallback)

**Date:** 2026-07-01
**Change package:** 260701_13
**Version:** 0.12.0
**Scope:** MONOLITH APK Analyzer platform development (not analysed apps).

## Goal / why

After v0.11.0 always boots an emulator, a real report still showed *"APK failed to install (check ABI / minSdk /
signature)"* — a generic message. The user (with a precise diagnosis) pointed at the #1 cause: **ABI mismatch**
(`INSTALL_FAILED_NO_MATCHING_ABIS`) — the emulator is x86_64 and the app ships ARM-only (esp. armeabi-v7a) native
libs. Requirement: the runner must diagnose the specific install failure, try fallbacks, and always give a concrete
remedial step — never just "failed"/"unsupported".

## Investigation

The booted emulator's abilist is **`x86_64, arm64-v8a`** (it translates arm64 via `libndk_translation.so`), so
arm64/x86_64 apps install (the universal test APK did → 14 screens). A **32-bit-ARM-only (armeabi-v7a) app** cannot
install on this image (armeabi-v7a is not in the abilist), and a same-id app signed with a different key fails on
signature. Those are the real failure classes to handle.

## Added / Fixed — `layers/dynamic_apk.py`

- **ABI inventory**: `_apk_abis()` reads the APK's `lib/<abi>/` set authoritatively; `_device_abis()` reads
  `ro.product.cpu.abilist`. Both recorded on `dynamic.apkAbis` / `dynamic.deviceAbis`.
- **Uninstall-first install**: `_try_install()` uninstalls any prior/mismatched-signature copy, then installs — this
  alone fixes the signature-mismatch class.
- **Specific diagnosis**: `_diagnose_install()` classifies the raw `adb install` output into
  `INSTALL_FAILED_NO_MATCHING_ABIS` (ABI mismatch, with the app's ABIs vs the device abilist) /
  `INSTALL_FAILED_OLDER_SDK` (minSdk > device API) / `INSTALL_FAILED_UPDATE_INCOMPATIBLE` (signature) /
  `INSTALL_FAILED_INSUFFICIENT_STORAGE` / `INSTALL_PARSE_FAILED`, each with a **concrete remediation** — the finding
  is never a generic "failed" (the `benchmark_id` carries the precise install code).
- **Compatible-emulator fallback**: on ABI mismatch the layer calls `ensure_device(required_abis=<apk abis>)` to
  obtain an ABI-matched emulator and retries the install once, recording `dynamic.emulator.abiFallback`.

## Added — `layers/emulator.py`

- `ensure_device(ctx, required_abis=None)` is now **ABI-aware**: it uses an attached device only if its abilist
  covers a required ABI; otherwise it picks a system-image ABI that can run the app (`_image_abi_for`: arm64-v8a→
  x86_64 image (translates), armeabi-v7a→x86 image, etc.), finds an **installed** matching image or **downloads**
  one via sdkmanager (`_install_image`, gated by `dynamic.allow_image_download`), creates an AVD (`monolith_<abi>`),
  boots it, and returns the serial. Fully bounded + honest (returns None + reason if it can't).

## Modified

- `config.py` — `dynamic.allow_image_download` (True). `dependencies.md` — ABI-compatibility + fallback notes.
  Version → **0.12.0**.

## Acceptance criteria (met)

- Install failure is diagnosed to a specific class with a concrete fix (verified via unit tests of `_diagnose_install`
  for all five classes). ✔

- ABI mismatch triggers a compatible-emulator attempt (image-ABI selection verified: arm64-v8a→x86_64, armeabi-v7a→

## Regression (verified)

`ensure_device(required_abis)` returns the compatible attached emulator; install + screencap succeed; engine +
dynamic + emulator import clean. All prior invariants (26 sections, 17-dim scorecard, 1000 QA, 100 corroborators,
163 benchmark checks) hold.

## Risk / rollback

Medium: the image-download fallback is heavy (large sdkmanager download) + slow (armeabi-v7a images are 32-bit) — it
is best-effort, gated (`allow_image_download`), and on failure MONOLITH reports the precise ABI mismatch + fix instead
of hanging. Rollback = restore the single-attempt install + generic finding.
