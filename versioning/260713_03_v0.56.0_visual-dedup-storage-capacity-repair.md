# Jupiter v0.56.0 — visual duplicate coverage and retail storage capacity repair

**Date:** 2026-07-13
**Branch:** `main`
**Scope:** regression-free repair of the reported `Similar photos (0)` false negative and 256 GB → 274.9 GB storage-total defect.

## User requirement

1. A large real image corpus must again be evaluated as visual duplicates; the app must not silently discard images because an earlier calculation was temporarily unavailable.
2. A phone sold as 256 GB must not be presented as 274.9 GB merely because the filesystem reports a binary 256 GiB allocation.
3. Existing exact duplicate detection, keeper protection, size filters, Select all/Deselect all, Recycle Bin workflow and arrival-alert behavior must not regress.

## Root cause and implemented requirement mapping

| Reported symptom | Root cause | Requirement-complete change |
| --- | --- | --- |
| `Similar photos (0)` despite many real photos | A nullable descriptor calculation was persisted as permanent `UNHASHABLE`; temporary provider/decoder/Room failures consequently removed a photo permanently from the Similar query. | Typed retry policy: only a concrete decoder refusal is permanent. Null is retryable. v7→v8 clears the incorrectly persisted sentinel for IMAGE rows once. |
| Opening Duplicate cleanup did not force photo analysis | The scheduler built expedited work with `BatteryNotLow`, an invalid WorkManager expedited constraint. The construction exception was swallowed. Disabled continuous indexing could also no-op an explicit UI request. | Background and explicit requests use separate valid requests. Explicit UI request is constraint-free expedited work with fallback and can run after a user asks to review duplicates. |
| Extensionless image ignored | Index classification used filename extension only. | MediaStore MIME type participates in classification; `image/*` becomes `FileType.IMAGE`. |
| 274.9 GB shown for 256 GB phone | Raw filesystem bytes were formatted directly as decimal GB. | Exact binary GiB allocation is normalized to its matching consumer retail decimal tier. Platform free bytes are retained; nonstandard OEM capacity stays platform-backed. |

## Invariants / non-regression contract

- Full `contentHash` remains the only exact deduplication decision; perceptual hash remains review-only and cannot authorize deletion.
- Existing quality-ranked keeper protection gates all selection modes, including Select all; only visible active scope is selected.
- `Long.MIN_VALUE` is a terminal state only after an actual decoder result declares the file unsupported. A null result cannot advance into this terminal state.
- The v7→v8 migration is narrow and idempotent in effect: it alters only IMAGE records carrying the faulty legacy sentinel. No index table is dropped and there is no destructive fallback.
- Capacity normalization changes only presentation of known raw binary allocation sizes. It does not fabricate category file data, alter platform free bytes, or force a guessed tier over a genuine nonstandard OEM report.
