# v0.13.0 — Recoverable Recycle Bin (safe delete + restore + audit)

**Date:** 2026-07-01
**Change package:** 260701_18
**Version:** 0.13.0
**Scope:** Jupiter Android app development (not managed device files).

## Goal / why

Port Sift's safe-delete: deletions must be **recoverable**, not permanent. The user asked to keep
mining `ultimate-duplicate-handler`; the desktop app's #1 data-safety feature is the quarantine/
Recycle Bin + restore. This makes every Jupiter delete reversible and adds an audit trail.

## Added

- **`data/trash/`** — `TrashEntry` (`@Entity`), `TrashDao`, `TrashDatabase` (own `jupiter_trash.db`,
  **non-destructive** so metadata survives), `TrashRepositoryImpl`; `domain/model/TrashItem.kt`;
  `domain/repository/TrashRepository.kt`; `di/TrashModule.kt` (`@Provides` DB+DAO, separate `@Binds`).
  Trash root = `context.getExternalFilesDir("trash")` — app-private and already excluded from all
  storage scans/index walks (no scanner change needed). Audit via `deletedAt`.
- **`feature/trash/`** — Recycle Bin screen (list with name/size/original-location/deleted-time;
  **Restore**, **Delete permanently**, **Empty** with confirm; honest empty-state) + VM + UiState.
- **Nav** — `Destination.Trash` + `JupiterNavHost` route + a "Recycle Bin" tile in More.

## Modified

- **`data/file/FileOperationsManager.kt`** — `delete()` routes every item through
  `TrashRepository.moveToTrash`; on failure the **source is preserved** (no hard-delete fallback),
  returning `Failure` so the UI can report it.
- **`feature/cleanup/DuplicatesViewModel.kt` + `SmartMergeViewModel.kt`** — the two direct
  `File.delete()` bypasses (flagged in the earlier bug-hunt) now `moveToTrash`, preserving their
  existing per-item feedback + selection/merge logic.

## Acceptance criteria (met)

- Deleting a file moves it to the Recycle Bin; it can be restored to its original location (or a
  non-clashing "name (restored)" if occupied) or permanently removed; Empty clears all. ✔
- A file that cannot be safely trashed is **preserved**, never destroyed. ✔
- No permanent `File.delete()` remains on the user-facing delete paths. ✔

## Regression checks (verified)

- Existing delete confirmations, selection, and dup/merge feedback preserved; existing nav
  destinations/tiles intact; the index/scan walks unaffected (trash lives under the excluded
  `Android/data` tree).
- Room codegen + Hilt provisioning validated (`@Provides` DB/DAO + `@Binds` repo; no cycle).

## Verification

- Workflow `w6vd527qf`: 3 build owners + 2 verifiers (one **data-loss-focused**).
- The data-loss verifier CAUGHT a real destruction path: in the cross-volume copy-then-delete, a
  partially-failed source delete then deleted the verified target — the only complete copy. **Fixed**
  in `TrashRepositoryImpl.movePath` (never delete a verified target; a partial source leftover is
  retained over losing data) and in the `moveToTrash` DB-rollback (moves the payload back out instead
  of deleting the target). The workflow had also left `movePath` truncated mid-function (non-compiling);
  rewritten whole.
- **CI (authoritative):** green run to be recorded on the babysit confirmation.

## Remaining risk / rollback

- Medium: delete-path rewrite is data-safety-critical; the cardinal "preserve on failure" rule is
  enforced in every branch and was adversarially reviewed + hand-verified (brace balance, imports,
  routing). CI is authoritative for compile. Rollback = revert this commit; deletes return to the
  (confirmed) hard-delete path.
- Planned next: exclude a visible `.JupiterTrash` name is unnecessary (app-private dir); add auto-purge
  of old trash; then scan filters + perceptual near-duplicate.

## Lessons captured this iteration

- **Directory `deleteRecursively()` can partially succeed and still return false** — after a verified
  cross-volume copy, never delete the target on a source-delete failure; retain the complete copy.
- **A dedicated "can this lose data?" verifier is worth it** on any delete-path change; it found a
  destruction scenario a generic reviewer missed. (Sync to the external lessons orchestrator.)
